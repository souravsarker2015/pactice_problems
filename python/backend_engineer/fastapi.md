নিচে পাইথন ব্যাকএন্ড ইন্টারভিউয়ের জন্য গুরুত্বপূর্ণ টপিকগুলো সহজভাবে আলোচনা করা হলো:

---

### ১. FastAPI vs Django: প্রধান পার্থক্য

| ফিচার | FastAPI | Django |
| :--- | :--- | :--- |
| **আর্কিটেকচার** | **Micro-framework:** খুব ছোট এবং মডুলার। শুধু যা প্রয়োজন তা-ই যোগ করা যায়। | **Monolithic:** সব কিছু বিল্ট-ইন থাকে (Admin, Auth, ORM)। |
| **গতি (Speed)** | অত্যন্ত দ্রুত (Go বা Node.js-এর সমতুল্য)। | তুলনামূলক স্লো কারণ এটি অনেক ভারী। |
| **Async সাপোর্ট** | নেটিভভাবে **Async/Await** এর ওপর ভিত্তি করে তৈরি। | মূলত সিনক্রোনাস, তবে এখন ধীরে ধীরে অ্যাসিঙ্ক সাপোর্ট যোগ হচ্ছে। |
| **ভ্যালিডেশন** | **Pydantic** ব্যবহার করে টাইপ সেফটি নিশ্চিত করে। | Django **Forms/Serializers** ব্যবহার করে। |

**সহজ কথায়:** যদি হাই-পারফরম্যান্স এবং আধুনিক মাইক্রোসার্ভিস দরকার হয়, তবে **FastAPI** সেরা। আর যদি বড় এবং কমপ্লেক্স অ্যাপ্লিকেশন দ্রুত (Admin panel সহ) তৈরি করতে হয়, তবে **Django** এগিয়ে।

---

### ২. FastAPI কেন এত Fast?

FastAPI-র গতির পেছনে তিনটি প্রধান কারণ রয়েছে:
1.  **Starlette:** এটি একটি অত্যন্ত হালকা ASGI ফ্রেমওয়ার্ক যা রাউটিং এবং ওয়েব রিলেটেড কাজগুলো খুব দ্রুত করে।
2.  **Pydantic:** ডাটা ভ্যালিডেশনের জন্য এটি পাইথনের সবচেয়ে দ্রুততম লাইব্রেরিগুলোর একটি। এটি টাইপ হিন্ট ব্যবহার করে ডাটা পার্স করে।
3.  **Asynchronous:** এটি নন-ব্লকিং। অর্থাৎ একটি রিকোয়েস্ট ডাটাবেসের জন্য অপেক্ষা করলে সেই সময়ে এটি অন্য রিকোয়েস্ট প্রসেস করতে পারে।

---

### ৩. ASGI কী? (Asynchronous Server Gateway Interface)

**ASGI** হলো পাইথন ওয়েব অ্যাপ এবং ওয়েব সার্ভারের মধ্যে যোগাযোগের একটি আধুনিক মাধ্যম। 
* এটি একটি অ্যাপ্লিকেশনকে একই সাথে একাধিক রিকোয়েস্ট (Concurrency) হ্যান্ডেল করার সুযোগ দেয়।
* এটি শুধু HTTP নয়, বরং **WebSockets** এবং **HTTP/2** এর মতো আধুনিক প্রোটোকলগুলোও সাপোর্ট করে।

---

### ৪. WSGI vs ASGI: পার্থক্য

ইন্টারভিউতে এটি একটি কমন প্রশ্ন। একটি অ্যানালজি দিয়ে বোঝা যাক:

* **WSGI (Synchronous):** মনে করুন একটি সরু গলি যেখানে একটি গাড়ি গেলে অন্যটি ঢুকতে পারে না। এখানে একটি রিকোয়েস্ট প্রসেস হওয়া পর্যন্ত অন্যটি অপেক্ষা করে (Blocking)।
* **ASGI (Asynchronous):** এটি একটি মাল্টি-লেন হাইওয়ের মতো। এখানে অনেকগুলো রিকোয়েস্ট একসাথে চলতে পারে। যদি কোনো রিকোয়েস্ট ডাটাবেস কলের জন্য থামে, তবে অন্য রিকোয়েস্টগুলো তাকে ওভারটেক করে চলে যেতে পারে (Non-blocking)।

| ফিচারের নাম | WSGI (যেমন: Django, Flask) | ASGI (যেমন: FastAPI, Sanic) |
| :--- | :--- | :--- |
| **রিকোয়েস্ট টাইপ** | শুধুমাত্র সিনক্রোনাস। | সিনক্রোনাস এবং অ্যাসিঙ্ক্রোনাস। |
| **Concurrency** | থ্রেড পুলের ওপর নির্ভরশীল। | ইভেন্ট লুপের ওপর ভিত্তি করে হাজারো কানেকশন হ্যান্ডেল করে। |
| **ব্যবহার** | ট্র্যাডিশনাল ওয়েব অ্যাপে। | রিয়েল-টাইম অ্যাপ বা হাই-ট্রাফিক এপিআই-তে। |

---

### ৫. Pydantic কী?

**Pydantic** হলো একটি পাইথন লাইব্রেরি যা ডাটা ভ্যালিডেশন এবং সেটিংস ম্যানেজমেন্টের জন্য ব্যবহৃত হয়।

* **কেন এটি দরকার?** পাইথন একটি ডাইনামিক ল্যাঙ্গুয়েজ। তাই কোনো ফাংশনে ভুল ডাটা পাঠালে রানটাইমে এরর হয়। Pydantic পাইথনের **Type Hints** ব্যবহার করে ইনপুট ডাটা চেক করে।
* **কাজ:** যদি আপনার এপিআই-তে `age` ফিল্ডে `int` আসার কথা থাকে কিন্তু কেউ `string` পাঠায়, তবে Pydantic সাথে সাথে এরর দিয়ে দিবে। এটি অটোমেটিক টাইপ কনভার্সনও করে (যেমন: `"123"` কে `123` বানিয়ে দেয়)।

**কোড উদাহরণ:**
```python
from pydantic import BaseModel

class User(BaseModel):
    id: int
    username: str
    email: str

# ডাটা ভুল থাকলে এখানেই এরর আসবে, ফাংশনের ভেতরে যাবে না।
```

### ইন্টারভিউ টিপ:
ইন্টারভিউতে **FastAPI** নিয়ে কথা বলার সময় **Swagger (OpenAPI)** ডকুমেন্টেশনের কথা উল্লেখ করতে ভুলবেন না। এটি এপিআই টেস্টিং এবং ফ্রন্টএন্ড ডেভেলপারদের সাথে কোলাবরেশন অনেক সহজ করে দেয়।

আপনি কি ডাটাবেস ইন্টিগ্রেশন (SQLAlchemy/Alembic) নিয়ে কিছু জানতে চান?

ইন্টারভিউতে টেকনিক্যাল বিষয়গুলো সহজে এবং সরাসরি বোঝানোর জন্য নিচের পয়েন্টগুলো ফলো করুন:

### ১. Dependency Injection (DI)
**এক কথায়:** এটি এমন একটি সিস্টেম যেখানে একটি ফাংশন বা রাউটের জন্য প্রয়োজনীয় "সার্ভিস" (যেমন: DB connection, Auth) বাইরে থেকে সরবরাহ করা হয়।

* **কেন ব্যবহার হয়?**: কোড রিপিটেশন কমাতে এবং টেস্টিং সহজ করতে। 
* **কীভাবে কাজ করে?**: FastAPI-তে `Depends()` ফাংশন ব্যবহার করে এটি করা হয়।
* **সুবিধা**: ধরুন আপনার ৫০টি এপিআই-তে ডাটাবেস কানেকশন লাগবে। আপনি ৫০ বার কোড না লিখে একবার লিখে তা `Depends()` দিয়ে ইনজেক্ট করবেন।



**ছোট উদাহরণ:**
```python
from fastapi import Depends, FastAPI

def get_db():
    return "DB Connection"

app = FastAPI()

@app.get("/")
def read_data(db = Depends(get_db)): # এখানে db ইনজেক্ট করা হয়েছে
    return {"data": db}
```

### ২. BackgroundTasks
**এক কথায়:** ক্লায়েন্টকে দ্রুত রেসপন্স পাঠিয়ে দিয়ে পেছনের সারিতে (Background) কোনো কাজ শেষ করার পদ্ধতি।

* **কখন ব্যবহার করবেন?**: এমন সব কাজ যা শেষ হতে সময় লাগে কিন্তু ইউজারের সেটার জন্য অপেক্ষা করার দরকার নেই। যেমন:
    * ইমেইল পাঠানো।
    * বড় ফাইল প্রসেস করা।
    * লগ ফাইল আপডেট করা।
* **কেন এটি দরকার?**: এটি এপিআই-কে সুপার ফাস্ট রাখে কারণ ইউজারকে ইমেইল পাঠানো শেষ হওয়া পর্যন্ত বসে থাকতে হয় না।



**ছোট উদাহরণ:**
```python
from fastapi import BackgroundTasks, FastAPI

app = FastAPI()

def send_email(email: str):
    # ইমেইল পাঠানোর লজিক (সময়সাপেক্ষ কাজ)
    print(f"Email sent to {email}")

@app.post("/signup")
async def signup(email: str, tasks: BackgroundTasks):
    tasks.add_task(send_email, email) # ব্যাকগ্রাউন্ডে কাজ শুরু হবে
    return {"message": "Signup successful, check email later!"} # রেসপন্স সাথে সাথে চলে যাবে
```

### ৩. ইন্টারভিউতে মনে রাখার মতো পার্থক্য
| বৈশিষ্ট্য | Dependency Injection | BackgroundTasks |
| :--- | :--- | :--- |
| **সময়** | রিকোয়েস্ট প্রসেস হওয়ার **আগে/চলাকালীন** কাজ করে। | রেসপন্স পাঠিয়ে দেওয়ার **পর** কাজ করে। |
| **উদ্দেশ্য** | কোড শেয়ারিং এবং লজিক ইনজেক্ট করা। | পারফরম্যান্স বাড়ানো এবং নন-ব্লকিং কাজ করা। |

**টিপ:** বড় কোনো প্রসেসিং (যেমন: ভিডিও কনভার্ট বা বাল্ক ইমেইল) হলে BackgroundTasks-এর বদলে **Celery + Redis** ব্যবহার করা ভালো। ইন্টারভিউয়ার যদি জিজ্ঞেস করে "কেন?", বলবেন—কারণ BackgroundTasks একই প্রসেসে চলে, আর Celery আলাদা ওয়ার্কার ব্যবহার করে।

পরবর্তী প্রশ্ন হিসেবে **FastAPI Middleware** বা **Pydantic Schemas** নিয়ে জানতে চান?

ইন্টারভিউতে টেকনিক্যাল গভীরতা বজায় রেখে ছোট এবং সরাসরি উত্তরগুলো নিচে দেওয়া হলো:

### ১. Middleware (মিডলওয়্যার)
**এক কথায়:** এটি এমন একটি ফাংশন যা প্রতিটি রিকোয়েস্ট আসার পর এবং রেসপন্স যাওয়ার আগে **মাঝপথে (Middle)** কাজ করে।

* **ফ্লো:** রিকোয়েস্ট → **Middleware** → রাউট ফাংশন → **Middleware** → রেসপন্স।
* **কেন ব্যবহার হয়?**: 
    * **Logging:** প্রতিটা রিকোয়েস্টের ডিটেইলস সেভ করা।
    * **CORS:** অন্য ডোমেইন থেকে রিকোয়েস্ট অ্যালাউ করা।
    * **Timing:** একটি এপিআই কল হতে কত সময় লাগছে তা মাপা।
    * **Authentication:** টোকেন ভ্যালিড কি না তা শুরুতেই চেক করা।

**উদাহরণ:**
```python
@app.middleware("http")
async def add_process_time_header(request, call_next):
    # রিকোয়েস্ট আসার সময় কাজ
    response = await call_next(request)
    # রেসপন্স যাওয়ার সময় কাজ
    response.headers["X-Process-Time"] = "0.01s"
    return response
```

### ২. Exception Handling (এরর হ্যান্ডলিং)
**এক কথায়:** কোডে কোনো ভুল বা এরর হলে সার্ভার ক্র্যাশ না করে ক্লায়েন্টকে সুন্দরভাবে (JSON আকারে) জানানো।

FastAPI-তে এটি মূলত দুটি উপায়ে করা হয়:

#### ক) HTTPException (স্ট্যান্ডার্ড উপায়)
যখন আপনি জানেন যে কোনো নির্দিষ্ট কারণে এরর হতে পারে (যেমন: ইউজার পাওয়া যায়নি)।
```python
from fastapi import HTTPException

if not user:
    raise HTTPException(status_code=404, detail="User not found")
```

#### খ) Custom Exception Handler (গ্লোবাল উপায়)
পুরো অ্যাপ্লিকেশনে কোনো নির্দিষ্ট টাইপের এরর হলে তা এক জায়গা থেকে হ্যান্ডেল করার জন্য এটি ব্যবহৃত হয়।

**উদাহরণ:**
```python
from fastapi.responses import JSONResponse

@app.exception_handler(ValueError) # যখনই কোথাও ValueError হবে
async def value_error_handler(request, exc):
    return JSONResponse(
        status_code=400,
        content={"message": "আপনার দেওয়া ইনপুট সঠিক নয়।"},
    )
```

### ৩. ইন্টারভিউ টিপ (সুইফট কম্পারিশন)

| ফিচার | Middleware | Exception Handler |
| :--- | :--- | :--- |
| **কখন চলে?** | সব রিকোয়েস্টের জন্য স্বয়ংক্রিয়ভাবে চলে। | শুধু যখন কোনো এরর ঘটে তখন চলে। |
| **মূল কাজ** | ডাটা মডিফিকেশন বা লগিং। | এরর মেসেজ কাস্টমাইজ করা। |
| **স্কোপ** | গ্লোবাল (পুরো অ্যাপে)। | নির্দিষ্ট এরর টাইপ অনুযায়ী। |

পরবর্তী কোনো টপিক যেমন **FastAPI Security (OAuth2/JWT)** বা **Alembic (Migration)** নিয়ে আলোচনা করব?

ইন্টারভিউতে টেকনিক্যাল পার্থক্যগুলো পরিষ্কারভাবে বোঝানোর জন্য নিচের পয়েন্টগুলো গুছিয়ে বলুন:

### ১. Async endpoint কেন ব্যবহার করবেন?
**এক কথায়:** হাই-ট্রাফিক হ্যান্ডেল করা এবং সার্ভারের রিসোর্স (CPU/RAM) বাঁচানোর জন্য।

* **Non-blocking I/O:** যখন কোনো রিকোয়েস্ট ডেটাবেস বা এক্সটার্নাল এপিআই কলের জন্য অপেক্ষা করে, তখন সার্ভার বসে না থেকে অন্য রিকোয়েস্ট প্রসেস করতে পারে।
* **Concurrency:** একটি সিঙ্গেল থ্রেড ব্যবহার করেই হাজার হাজার কানেকশন একসাথে ম্যানেজ করা যায়।
* **Resource Efficiency:** মাল্টি-থ্রেডিংয়ের তুলনায় মেমোরি অনেক কম খরচ হয়।

> **ইন্টারভিউ টিপ:** যদি আপনার এপিআই শুধু কম্পিউটেশনাল কাজ (যেমন: ইমেজ প্রসেসিং বা ক্যালকুলেশন) করে, তবে `async` ব্যবহার করে খুব একটা লাভ নেই। কিন্তু যদি I/O কাজ (DB, File, API) বেশি থাকে, তবে `async` মাস্ট।

---

### ২. Sync vs Async DB Call
ডেটাবেস কলের ক্ষেত্রে এই পার্থক্যটি সবচেয়ে বেশি চোখে পড়ে।



#### ক) Sync DB Call (ব্লকিং)
* **কাজ:** কোড যখন ডেটাবেস কুয়েরি পাঠায়, তখন উত্তর না আসা পর্যন্ত পুরো লাইন আটকে থাকে।
* **সমস্যা:** যদি ১০০০ জন ইউজার একসাথে রিকোয়েস্ট পাঠায় এবং আপনার সার্ভারে মাত্র ১০০টি থ্রেড থাকে, তবে বাকি ৯০০ জন ইউজার "Waiting" মোডে থাকবে।
* **উদাহরণ:** Standard `psycopg2` বা Django-র ডিফল্ট ORM।

#### খ) Async DB Call (নন-ব্লকিং)
* **কাজ:** কোড কুয়েরি পাঠিয়ে দিয়ে ইভেন্ট লুপকে বলে— "আমি ওয়েট করছি, তুমি অন্য কাজ করো।" উত্তর আসলে লুপ আবার এই কোডটি রান করে।
* **সুবিধা:** একই সময়ে অনেক বেশি ইউজারকে সার্ভিস দেওয়া যায় কারণ কোনো থ্রেডই ডেটাবেসের জন্য অলস বসে থাকে না।
* **উদাহরণ:** `SQLAlchemy` (with `asyncio`), `Tortoise ORM`, বা `Motor` (MongoDB)।

---

### এক নজরে তুলনা

| বৈশিষ্ট্য | Sync DB Call | Async DB Call |
| :--- | :--- | :--- |
| **থ্রেড ব্যবহার** | প্রতিটি রিকোয়েস্টের জন্য আলাদা থ্রেড লাগে। | একটি সিঙ্গেল থ্রেডেই অনেক কুয়েরি চলে। |
| **পারফরম্যান্স** | লো-ট্রাফিকে ভালো, হাই-ট্রাফিকে স্লো। | হাই-ট্রাফিকে অত্যন্ত ফাস্ট। |
| **লাইব্রেরি** | `requests`, `psycopg2` | `httpx`, `asyncpg`, `motor` |
| **জটিলতা** | কোড লেখা সহজ। | কোড স্ট্রাকচার (await) মেইনটেইন করা জরুরি। |

### ৩. ইন্টারভিউয়ার যদি জিজ্ঞেস করে: "কখন কোনটা ব্যবহার করবেন?"
আপনার উত্তর হওয়া উচিত—
* "যদি প্রোজেক্টটি ছোট হয় এবং ট্রাফিক কম থাকে, তবে **Sync** সহজ ও নিরাপদ।"
* "যদি এমন সিস্টেম হয় যেখানে প্রতি সেকেন্ডে হাজার হাজার ডাটাবেস হিট হবে বা থার্ড-পার্টি এপিআই কল করতে হবে, তবে **Async** ব্যবহার করাই বুদ্ধিমানের কাজ।"

পরবর্তী টপিক হিসেবে **JWT Authentication** বা **FastAPI-তে Pydantic Model vs Schema** নিয়ে আলোচনা করব?

ইন্টারভিউতে **SQLAlchemy** এবং **Connection Pooling** নিয়ে নিচের পয়েন্টগুলো সংক্ষেপে এবং স্পষ্টভাবে বলতে পারেন:

### ১. SQLAlchemy Use Case
**SQLAlchemy** হলো পাইথনের সবচেয়ে জনপ্রিয় **ORM (Object Relational Mapper)**। এর প্রধান কাজগুলো হলো:

* **Object-Database Mapping:** ডেটাবেস টেবিলকে পাইথন **Class** এবং রো (row)-কে **Object** হিসেবে ট্রিট করা। এতে SQL না লিখে পাইথন কোড দিয়ে কুয়েরি করা যায়।
* **Database Agnostic:** একই কোড ব্যবহার করে আপনি MySQL, PostgreSQL বা SQLite-এ কাজ করতে পারেন। শুধু কানেকশন স্ট্রিং পরিবর্তন করলেই হয়।
* **Complex Queries:** জয়েন (Join), সাব-কুয়েরি বা অ্যাগ্রিগেশন সহজে ম্যানেজ করা যায়।
* **Security:** এটি অটোমেটিক **SQL Injection** প্রতিরোধ করে।
* **Alembic Integration:** ডেটাবেস স্কিমা পরিবর্তন বা মাইগ্রেশন ট্র্যাক করার জন্য এটি বেস্ট।

---

### ২. Connection Pooling
**এক কথায়:** ডেটাবেসের সাথে বারবার কানেকশন তৈরি এবং বন্ধ না করে, কিছু কানেকশন আগে থেকেই তৈরি করে একটি **"পুলে"** জমা রাখা।



#### এটি কেন দরকার?
ডেটাবেসের সাথে নতুন কানেকশন তৈরি করা একটি **"Expensive Process"** (সময় এবং রিসোর্স বেশি লাগে)।
* **Efficiency:** যখনই কোনো এপিআই রিকোয়েস্ট আসে, সে পুল থেকে একটি ফ্রি কানেকশন নিয়ে কাজ করে এবং কাজ শেষে আবার পুলে ফেরত দেয়।
* **Performance:** কানেকশন হ্যান্ডশেকিংয়ের সময় বেঁচে যায়, ফলে এপিআই ফাস্ট হয়।
* **Traffic Management:** ডেটাবেস যেন অতিরিক্ত কানেকশনে ক্র্যাশ না করে, তাই পুলের মাধ্যমে সর্বোচ্চ কানেকশন সংখ্যা নিয়ন্ত্রণ করা যায়।

---

### ৩. SQLAlchemy-তে কানেকশন পুলিং
SQLAlchemy ডিফল্টভাবেই কানেকশন পুলিং হ্যান্ডেল করে। `create_engine` করার সময় কিছু প্যারামিটার সেট করা যায়:

```python
from sqlalchemy import create_engine

engine = create_engine(
    "postgresql://user:pass@localhost/db",
    pool_size=10,         # পুলে কয়টি কানেকশন থাকবে
    max_overflow=20,      # চাপের মুখে অতিরিক্ত কয়টি কানেকশন খোলা যাবে
    pool_timeout=30,      # কানেকশনের জন্য কতক্ষণ অপেক্ষা করবে
    pool_recycle=1800     # কতক্ষণ পর কানেকশন রিসেট হবে
)
```

### ৪. ইন্টারভিউ টিপ (Comparison)

| বৈশিষ্ট্য | সাধারণ কানেকশন (No Pool) | কানেকশন পুলিং (With Pool) |
| :--- | :--- | :--- |
| **গতি** | স্লো (প্রতিবার নতুন কানেকশন লাগে)। | ফাস্ট (আগে থেকে তৈরি কানেকশন ব্যবহার হয়)। |
| **রিসোর্স** | CPU/RAM খরচ বেশি। | অপ্টিমাইজড। |
| **লিমিট** | অনেক ইউজার একসাথে আসলে DB ক্র্যাশ করতে পারে। | নির্দিষ্ট লিমিটের মধ্যে সেফলি কাজ করে। |

পরবর্তী টপিক হিসেবে কি **SQLAlchemy Async Session** নাকি **Alembic Migration** নিয়ে আলোচনা করব?

ইন্টারভিউতে ডাটাবেস এবং সিকিউরিটি নিয়ে এই দুটি প্রশ্ন খুবই কমন। নিচে পয়েন্ট আকারে দেওয়া হলো:

### ১. Transaction Handling

**এক কথায়:** ডাটাবেসে একাধিক কাজকে একটি ইউনিটে পরিণত করা। হয় সব কাজ সফল হবে, না হয় কিছুই হবে না।

  * **কেন দরকার?**: ধরুন একজন গ্রাহক টাকা পাঠালেন। তার অ্যাকাউন্ট থেকে টাকা কাটলো কিন্তু যান্ত্রিক ভুলের কারণে যাকে পাঠালেন তার অ্যাকাউন্টে যোগ হলো না। ট্রানজেকশন হ্যান্ডলিং এই সমস্যা সমাধান করে।
  * **ACID Properties:** ট্রানজেকশন এই চারটি নিয়ম মেনে চলে— Atomicity, Consistency, Isolation, Durability।
  * **SQLAlchemy-তে ব্যবহার:** `commit()` দিলে সব সেভ হয়, আর এরর হলে `rollback()` দিয়ে আগের অবস্থায় ফিরে যাওয়া যায়।

**কোড উদাহরণ:**

```python
async with async_session() as session:
    async with session.begin(): # ট্রানজেকশন শুরু
        try:
            # Step 1: টাকা কমানো
            # Step 2: টাকা বাড়ানো
            await session.commit()
        except:
            await session.rollback() # এরর হলে সব বাতিল
            raise
```

### ২. Authentication in FastAPI

FastAPI-তে অথেন্টিকেশন সাধারণত **JWT (JSON Web Token)** এবং **OAuth2** দিয়ে করা হয়।

#### ক) ফ্লো (Workflow):

1.  **Login:** ইউজার তার ইউজারনেম এবং পাসওয়ার্ড পাঠায়।
2.  **Validation:** ব্যাকএন্ডে পাসওয়ার্ড চেক করা হয় (Hashed password)।
3.  **Token Generation:** পাসওয়ার্ড মিললে একটি **JWT Token** তৈরি করে ক্লায়েন্টকে দেওয়া হয়।
4.  **Authorized Request:** পরবর্তী প্রতিটা রিকোয়েস্টে ইউজার এই টোকেনটি হেডার হিসেবে পাঠায়।
5.  **Verification:** ব্যাকএন্ড টোকেনটি ডিকোড করে ইউজারের পরিচয় নিশ্চিত করে।

#### খ) মূল কম্পোনেন্টসমূহ:

  * **OAuth2PasswordBearer:** এটি এপিআই-এর হেডারে "Authorization: Bearer \<token\>" আছে কি না তা চেক করে।
  * **PyJWT/Jose:** টোকেন এনকোড এবং ডিকোড করার জন্য।
  * **Passlib (bcrypt):** পাসওয়ার্ড হ্যাশ করার জন্য (পাসওয়ার্ড কখনো ডাটাবেসে প্লেইন টেক্সট হিসেবে সেভ করতে হয় না)।

| বিষয় | মূল উদ্দেশ্য | মনে রাখার মতো পয়েন্ট |
| :--- | :--- | :--- |
| **Transaction** | ডাটার নির্ভুলতা (Consistency) রাখা। | Commit এবং Rollback। |
| **Authentication** | ইউজারের পরিচয় নিশ্চিত করা। | JWT, OAuth2, এবং Hashed Password। |

### ইন্টারভিউ টিপ:

ট্রানজেকশন নিয়ে কথা বলার সময় **Context Manager** (`with` statement) ব্যবহারের কথা উল্লেখ করবেন, কারণ এটি অটোমেটিক কানেকশন ক্লোজ এবং এরর হ্যান্ডলিং সহজ করে দেয়। আর অথেন্টিকেশনের ক্ষেত্রে বলবেন যে আপনি পাসওয়ার্ড সেভ করার জন্য **bcrypt** বা **Argon2** অ্যালগরিদম পছন্দ করেন।

ইন্টারভিউয়ের আগে আর কোনো নির্দিষ্ট টপিক যেমন **Testing (Pytest)** বা **Dockerization** নিয়ে রিভিশন দিতে চান?


ইন্টারভিউতে OAuth2 এবং JWT নিয়ে প্রশ্ন করলে নিচের মতো করে গুছিয়ে উত্তর দিতে পারেন:

### ১. OAuth2 in FastAPI
**OAuth2** হলো একটি **Authorization Framework** বা প্রোটোকল। এটি মূলত একটি নিয়মাবলী যা ঠিক করে দেয় একজন ইউজার কীভাবে লগইন করবেন এবং অ্যাপটি কীভাবে তাকে অ্যাক্সেস দেবে।

* **FastAPI-তে সুবিধা:** FastAPI-তে `fastapi.security` মডিউলের মাধ্যমে `OAuth2PasswordBearer` বিল্ট-ইন থাকে। এটি স্বয়ংক্রিয়ভাবে ব্রাউজারের Swagger UI-তে একটি "Authorize" বাটন যোগ করে দেয়।
* **Scopes:** OAuth2 এর মাধ্যমে আপনি ইউজারের পারমিশন লেভেল (যেমন: Read, Write) আলাদা করতে পারেন।

### ২. JWT (JSON Web Token) in FastAPI
**JWT** হলো তথ্যের একটি এনকোড করা স্ট্রাকচার যা সার্ভার এবং ক্লায়েন্টের মধ্যে নিরাপদে তথ্য আদান-প্রদান করে। এটি মূলত OAuth2 প্রোটোকলের মধ্যে একটি **Token Type** হিসেবে ব্যবহৃত হয়।

**JWT এর তিনটি অংশ থাকে:**
1.  **Header:** অ্যালগরিদম টাইপ।
2.  **Payload:** ইউজারের তথ্য (যেমন: `user_id`, `expire_time`) থাকে।
3.  **Signature:** টোকেনটি কেউ টেম্পার (Tamper) করেছে কি না তা চেক করার জন্য ব্যবহৃত হয়।

---

### ৩. OAuth2 এবং JWT কীভাবে একসাথে কাজ করে?
ইন্টারভিউতে এই ধাপগুলো বলা জরুরি:

1.  **Request:** ইউজার তার ইউজারনেম এবং পাসওয়ার্ড দিয়ে লগইন রিকোয়েস্ট পাঠায়।
2.  **Verify:** সার্ভার ডাটাবেস চেক করে পাসওয়ার্ড সঠিক কি না তা নিশ্চিত করে।
3.  **Issue Token:** পাসওয়ার্ড ঠিক থাকলে সার্ভার একটি **JWT Token** জেনারেট করে ক্লায়েন্টকে পাঠিয়ে দেয়।
4.  **Store:** ক্লায়েন্ট (Frontend/Mobile) সেই টোকেনটি লোকাল স্টোরেজ বা কুকিতে সেভ করে রাখে।
5.  **Access:** পরবর্তী রিকোয়েস্টে ক্লায়েন্ট হেডার হিসেবে `Authorization: Bearer <token>` পাঠায়।
6.  **Validate:** FastAPI-র `Depends()` মেকানিজম ওই টোকেনটি ডিকোড করে ইউজারের আইডেন্টিটি ভ্যালিডেট করে।

| বৈশিষ্ট্য | OAuth2 | JWT |
| :--- | :--- | :--- |
| **ধরন** | এটি একটি প্রোটোকল বা ফ্রেমওয়ার্ক। | এটি একটি ডাটা ফরম্যাট (Token)। |
| **কাজ** | ঠিক করে কীভাবে ইউজার পারমিশন পাবে। | এটি ইউজারের পরিচয় বহন করে। |
| **স্টেটলেসনেস** | সার্ভার-সাইড সেশন হতে পারে। | এটি পুরোপুরি **Stateless** (সার্ভারে সেভ করতে হয় না)। |

### ৫. সিকিউরিটি টিপস (Interview Points):
* **Secret Key:** JWT এনকোড করার জন্য একটি স্ট্রং সিক্রেট কি ব্যবহার করা উচিত।
* **Expiration:** টোকেনের মেয়াদ (Exp time) সবসময় কম রাখা উচিত এবং প্রয়োজনে **Refresh Token** ব্যবহার করা উচিত।
* **HTTPS:** টোকেন চুরি রোধে সবসময় HTTPS ব্যবহার করতে হবে।



আপনার ইন্টারভিউয়ের জন্য আর কোনো স্পেসিফিক টপিক (যেমন: **CORS**, **Pydantic nested models**) বাকি আছে? শুভকামনা!

ইন্টারভিউতে ফাইল হ্যান্ডলিং এবং স্ট্রিমিং নিয়ে প্রশ্ন করলে নিচের পয়েন্টগুলো সংক্ষেপে বলতে পারেন:

### ১. File Upload in FastAPI
FastAPI-তে ফাইল আপলোড করার জন্য দুটি প্রধান উপায় আছে:

* **`File`:** এটি ছোট ফাইলের জন্য ভালো। এটি পুরো ফাইলটি মেমোরিতে (RAM) লোড করে।
* **`UploadFile`:** এটি বড় ফাইলের জন্য সেরা। এটি **Python SpooledTemporaryFile** ব্যবহার করে, অর্থাৎ ফাইলটি মেমোরি লিমিট ক্রস করলে স্বয়ংক্রিয়ভাবে ডিস্কে সেভ হয়।

**কেন `UploadFile` ব্যবহার করবেন?**
1.  এটি ফাইলের মেটাডেটা (Filename, Content-Type) সহজে দেয়।
2.  এটি ফাইল অবজেক্টের মতো কাজ করে (`read`, `write`, `seek`, `close` মেথড আছে)।
3.  মেমোরি সেভ করে।

**কোড উদাহরণ:**
```python
from fastapi import UploadFile, File

@app.post("/upload")
async def upload_file(file: UploadFile = File(...)):
    contents = await file.read()
    return {"filename": file.filename}
```

### ২. StreamingResponse
**এক কথায়:** যখন কোনো বড় ডাটা বা ফাইল একবারে রেসপন্স হিসেবে না পাঠিয়ে ছোট ছোট টুকরো (Chunks) করে পাঠানো হয়, তাকে স্ট্রিমিং রেসপন্স বলে।

**কেন ব্যবহার করবেন?**
* **মেমোরি ম্যানেজমেন্ট:** যদি আপনার সার্ভার থেকে ৫ জিবি-র একটি ফাইল পাঠাতে হয়, তবে পুরোটা মেমোরিতে নিয়ে পাঠালে সার্ভার ক্র্যাশ করতে পারে। স্ট্রিমিং করলে অল্প অল্প করে ডাটা পাঠানো যায়।
* **রিয়েল-টাইম ডাটা:** ভিডিও স্ট্রিমিং বা জেনারেটিভ এআই (যেমন ChatGPT-র মতো লেখা আসা) এর ক্ষেত্রে এটি ব্যবহৃত হয়।

**কোড উদাহরণ:**
```python
from fastapi.responses import StreamingResponse

async def fake_video_streamer():
    for i in range(10):
        yield f"frame {i}\n" # জেনারেটর ফাংশন

@app.get("/video")
async def video_endpoint():
    return StreamingResponse(fake_video_streamer(), media_type="text/plain")
```

### ইন্টারভিউ টিপ:
ফাইল আপলোডের ক্ষেত্রে সবসময় উল্লেখ করবেন যে— *"বড় ফাইল হ্যান্ডেল করার সময় আমরা `shutil` লাইব্রেরি ব্যবহার করে ফাইলটি সরাসরি ডিস্কে স্ট্রীম করি যেন মেমোরি ওভারলোড না হয়।"* আর স্ট্রিমিং নিয়ে বললে বলবেন এটি **Iterator** বা **Generator** প্রোটোকল ব্যবহার করে কাজ করে।

আপনার ইন্টারভিউয়ের জন্য শুভকামনা! আশা করি সবগুলো টপিক আপনার কাজে আসবে। আর কোনো কনফিউশন আছে?


ইন্টারভিউতে পারফরম্যান্স এবং সিকিউরিটি নিয়ে এই দুটি টপিক খুবই অ্যাডভান্সড লেভেলের। নিচে পয়েন্ট আকারে সাজানো হলো:

### ১. FastAPI Performance Optimization
FastAPI এমনিতেই দ্রুত, কিন্তু প্রোডাকশন লেভেলে একে আরও অপ্টিমাইজ করার জন্য নিচের টেকনিকগুলো ব্যবহার করা হয়:

* **Uvicorn with Workers:** সিঙ্গেল প্রসেসের বদলে একাধিক ওয়ার্কার ব্যবহার করা। সাধারণত `(2 x cores) + 1` ফর্মুলা ব্যবহার হয়। 
    * *কমান্ড:* `uvicorn main:app --workers 4`
* **Use `ujson` or `orjson`:** FastAPI ডিফল্ট `json` ব্যবহার করে। কিন্তু `orjson` অনেক দ্রুত ডেটা এনকোড/ডিকোড করতে পারে।
* **Connection Pooling:** ডেটাবেস কানেকশন বারবার তৈরি না করে একটি কানেকশন পুল (SQLAlchemy-র মাধ্যমে) ব্যবহার করা।
* **Background Tasks:** ভারী কাজগুলো (যেমন ইমেইল পাঠানো বা রিপোর্ট জেনারেট) মেইন রিকোয়েস্ট লুপে না করে `BackgroundTasks` বা `Celery`-তে পাঠিয়ে দেওয়া।
* **Caching:** ঘন ঘন প্রয়োজন হয় এমন ডেটা (যেমন ইউজার প্রোফাইল বা কনফিগ) **Redis**-এ ক্যাশ করে রাখা।
* **Dependency Injection Overheads:** অপ্রয়োজনীয় কমপ্লেক্স ডিপেনডেন্সি এড়িয়ে চলা।

### ২. Rate Limiting
**এক কথায়:** একজন নির্দিষ্ট ইউজার বা IP অ্যাড্রেস নির্দিষ্ট সময়ে কতবার এপিআই কল করতে পারবে তা সীমাবদ্ধ করে দেওয়া। এটি মূলত **DDoS Attack** এবং **Brute Force** থেকে সার্ভারকে বাঁচায়।



**কীভাবে ইমপ্লিমেন্ট করবেন?**
FastAPI-তে রেট লিমিটিং করার সবচেয়ে জনপ্রিয় লাইব্রেরি হলো `slowapi` (যা Flask-Limiter এর মতো কাজ করে)।

* **Fixed Window:** এক মিনিটে ১০টি রিকোয়েস্ট। ১ মিনিট শেষ হলে আবার ১০টি।
* **Sliding Window:** আরও নিখুঁতভাবে রিকোয়েস্ট ট্র্যাক করা।
* **Storage:** রেট লিমিটের হিসাব রাখার জন্য **Redis** ব্যবহার করা সবচেয়ে ভালো, কারণ এটি অত্যন্ত ফাস্ট।

**ছোট উদাহরণ (SlowAPI):**
```python
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)
app.state.limiter = limiter

@app.get("/heavy-api")
@limiter.limit("5/minute") # প্রতি মিনিটে মাত্র ৫ বার কল করা যাবে
async def homepage(request: Request):
    return {"message": "Success"}
```

ইন্টারভিউয়ের জন্য এই দুটি অ্যাডভান্সড টপিক সংক্ষেপে নিচে দেওয়া হলো:

### ১. Caching in FastAPI
**এক কথায়:** বারবার ডাটাবেস বা এপিআই কল না করে, আগের পাওয়া রেজাল্ট কোনো দ্রুতগতির মেমোরিতে (যেমন **Redis**) জমিয়ে রাখা।

* **কেন দরকার?**: রেসপন্স টাইম কমানোর জন্য এবং ডাটাবেসের ওপর চাপ কমানোর জন্য।
* **কীভাবে কাজ করে?**: যখন কোনো রিকোয়েস্ট আসে, সার্ভার প্রথমে ক্যাশ চেক করে। থাকলে সেখান থেকেই ডেটা দিয়ে দেয় (Cache Hit)। না থাকলে ডাটাবেস থেকে এনে ক্যাশে সেভ করে রাখে (Cache Miss)।

**জনপ্রিয় লাইব্রেরি:** `fastapi-cache` বা সরাসরি `redis-py`।



**উদাহরণ (In-memory caching):**
```python
from fastapi_cache import FastAPICache
from fastapi_cache.decorator import cache

@app.get("/expensive-data")
@cache(expire=60) # ৬০ সেকেন্ডের জন্য ডেটা ক্যাশ হয়ে থাকবে
async def get_data():
    # অনেক সময়সাপেক্ষ কোনো কাজ বা DB query
    return {"data": "This is cached"}
```

### ২. Microservice Communication
মাইক্রোসার্ভিস আর্কিটেকচারে সার্ভিসগুলো একে অপরের সাথে মূলত দুইভাবে কথা বলে:

#### ক) Synchronous (সরাসরি যোগাযোগ)
* **পদ্ধতি:** HTTP/REST বা gRPC।
* **কাজ:** সার্ভিস A রিকোয়েস্ট পাঠিয়ে সার্ভিস B-এর উত্তরের জন্য অপেক্ষা করে।
* **সুবিধা:** সহজ এবং রিয়েল-টাইম।
* **অসুবিধা:** সার্ভিস B ডাউন থাকলে সার্ভিস A-ও কাজ করতে পারে না (Cascading Failure)।

#### খ) Asynchronous (মেসেজ ভিত্তিক যোগাযোগ)
* **পদ্ধতি:** Message Broker (যেমন: **RabbitMQ**, **Apache Kafka**)।
* **কাজ:** সার্ভিস A একটি মেসেজ বা ইভেন্ট ব্রোকারে পাঠিয়ে দেয় এবং নিজের কাজ চালিয়ে যায়। সার্ভিস B সময়মতো সেটা প্রসেস করে।
* **সুবিধা:** হাইলি স্কেলেবল এবং একটি সার্ভিস ডাউন থাকলেও সিস্টেম ভেঙে পড়ে না।

### ইন্টারভিউ কম্পারিজন টেবিল

| বৈশিষ্ট্য | Caching | Microservice Communication |
| :--- | :--- | :--- |
| **মূল টুল** | Redis, Memcached | RabbitMQ, Kafka, gRPC |
| **উদ্দেশ্য** | স্পিড বাড়ানো। | সার্ভিসের মধ্যে ডাটা আদান-প্রদান। |
| **সবচেয়ে বড় সুবিধা** | ডাটাবেস লোড কমায়। | সিস্টেমকে ডিকাপল (Decouple) করে। |

### ইন্টারভিউ টিপ:
যদি জিজ্ঞেস করে **Redis** কেন ক্যাশিংয়ের জন্য সেরা, বলবেন— এটি একটি **In-memory data structure**, তাই এটি ডিস্ক-ভিত্তিক ডাটাবেসের চেয়ে বহুগুণ ফাস্ট। আর মাইক্রোসার্ভিসের ক্ষেত্রে বলবেন যে, ট্রানজেকশনাল কাজের জন্য **Synchronous** আর ইভেন্ট-ড্রিভেন কাজের জন্য **Asynchronous** কমিউনিকেশন ভালো।

আপনার প্রিপারেশন বেশ সলিড! আর কি কোনো টপিক রিভিশন দেবেন, নাকি আপনি ইন্টারভিউয়ের জন্য প্রস্তুত?

ইন্টারভিউতে যখন সিস্টেম ডিজাইন বা স্কেলেবিলিটি নিয়ে কথা হয়, তখন এই দুটি টপিক খুবই গুরুত্বপূর্ণ। নিচে কনসাইজ উত্তরগুলো দেওয়া হলো:

### ১. FastAPI + Celery
**এক কথায়:** FastAPI যদি হয় দোকানের ফ্রন্ট ডেস্ক, তবে Celery হলো পেছনের বড় রান্নাঘর (Kitchen)। খুব ভারী বা সময়সাপেক্ষ কাজগুলো (যেমন: বাল্ক ইমেইল পাঠানো, ইমেজ প্রসেসিং, বড় রিপোর্ট জেনারেট করা) এপিআই থেকে আলাদা করে ব্যাকগ্রাউন্ডে করার জন্য Celery ব্যবহার করা হয়।

* **কেন দরকার?**: FastAPI-র `BackgroundTasks` ছোট কাজের জন্য ঠিক আছে, কিন্তু কাজ যদি কয়েক মিনিট সময় নেয় বা অনেক বেশি রিসোর্স লাগে, তবে `Celery` ব্যবহার করা হয় কারণ এটি আলাদা প্রসেস বা সার্ভারে রান করতে পারে।
* **Message Broker**: Celery নিজে একা কাজ করতে পারে না, তার একজন মিডলম্যান লাগে যাকে **Broker** বলা হয় (যেমন: **Redis** বা **RabbitMQ**)। এটি কাজগুলো কিউ (Queue) করে রাখে।



**ওয়ার্কফ্লো:**
1.  ইউজার রিকোয়েস্ট পাঠালো।
2.  FastAPI ব্রোকারকে (Redis) বললো— "এই কাজটা পরে করে দিও।"
3.  FastAPI সাথে সাথে ইউজারকে `Task ID` পাঠিয়ে দিলো।
4.  Celery Worker ব্রোকার থেকে কাজটা তুলে নিয়ে ব্যাকগ্রাউন্ডে শেষ করলো।

### ২. Dockerize FastAPI
**এক কথায়:** আপনার এপিআই এবং এর সাথে যা কিছু লাগে (Python version, libraries, OS settings) সব কিছু একটি "কন্টেইনার" বা প্যাকেজে বন্দি করা, যাতে এটি যেকোনো পিসি বা সার্ভারে একইভাবে রান করে।

**Dockerfile এর প্রধান স্টেপগুলো:**
* **Base Image:** সাধারণত `python:3.11-slim` বা `python:3.12-alpine` ব্যবহার করা হয় (হালকা সাইজের জন্য)।
* **Working Directory:** কন্টেইনারের ভেতরে কোড কোথায় থাকবে তা ঠিক করা (`/app`)।
* **Install Dependencies:** `requirements.txt` কপি করে `pip install` করা।
* **Expose Port:** সাধারণত পোর্ট `8000` খোলা রাখা।
* **CMD:** সার্ভার স্টার্ট করা (`uvicorn main:app --host 0.0.0.0 --port 8000`)।



**ছোট উদাহরণ (Dockerfile):**
```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### ৪. ইন্টারভিউ টিপ:
ইন্টারভিউয়ার যদি জিজ্ঞেস করে "Celery কেন ব্যবহার করব?", বলবেন: 
> *"যেকোনো I/O bound বা CPU intensive কাজ যা এপিআই রেসপন্স টাইম স্লো করে দিতে পারে, তা ব্যাকগ্রাউন্ডে অফলোড করার জন্য Celery ব্যবহার করি। আর Docker ব্যবহারের মূল কারণ হলো 'Production parity' নিশ্চিত করা—অর্থাৎ আমার পিসিতে যা চলছে, সার্ভারেও ঠিক তাই চলবে।"*

আপনি কি **Docker Compose** (একাধিক সার্ভিস যেমন FastAPI + Postgres + Redis একসাথে চালানো) নিয়ে কিছু জানতে চান? ইন্টারভিউয়ের জন্য শুভকামনা!


ইন্টারভিউতে প্রোডাকশন লেভেল ডেপ্লয়মেন্ট নিয়ে প্রশ্ন করলে এই বিষয়গুলো খুবই গুরুত্বপূর্ণ। নিচে কনসাইজ উত্তর দেওয়া হলো:

### ১. Deploy FastAPI
FastAPI ডেপ্লয় করার জন্য সাধারণ ধাপগুলো হলো:
1.  **Environment Setup:** পাইথন এবং প্রজেক্টের ডিপেনডেন্সি (requirements.txt) ইনস্টল করা।
2.  **Server:** সরাসরি Uvicorn ব্যবহার না করে প্রোডাকশন-গ্রেড সার্ভার (যেমন Gunicorn বা Docker) ব্যবহার করা।
3.  **Reverse Proxy:** সার্ভারের সামনে **Nginx** বা **Apache** রাখা। এটি সিকিউরিটি, SSL হ্যান্ডলিং এবং লোড ব্যালেন্সিং করে।
4.  **Process Manager:** সার্ভার ক্র্যাশ করলে যেন অটোমেটিক রিস্টার্ট হয়, তার জন্য **Systemd** বা **Supervisor** ব্যবহার করা।

### ২. Gunicorn + Uvicorn কেন ব্যবহার করবেন?
অনেকেই জিজ্ঞেস করে, "Uvicorn তো আছেই, তবে Gunicorn কেন?"

* **Uvicorn:** এটি একটি **ASGI server**। এটি পাইথন কোডকে এসিনক্রোনাসলি রান করতে পারে, কিন্তু এটি প্রসেস ম্যানেজমেন্টে (যেমন: মাল্টি-ওয়ার্কার হ্যান্ডলিং) খুব একটা শক্তিশালী নয়।
* **Gunicorn:** এটি একটি **Process Manager**। এটি একাধিক "Worker" তৈরি করতে পারে। যদি একটি ওয়ার্কার কোনো কারণে ক্র্যাশ করে, Gunicorn সাথে সাথে নতুন একটি ওয়ার্কার চালু করে দেয়।

**কম্বিনেশন:** প্রোডাকশনে আমরা Gunicorn-কে **Manager** হিসেবে এবং Uvicorn-কে **Worker class** হিসেবে ব্যবহার করি।



**রান করার কমান্ড:**
```bash
gunicorn -w 4 -k uvicorn.workers.UvicornWorker main:app
```
*(এখানে `-w 4` মানে ৪টি ওয়ার্কার প্রসেস চলবে যা একই সাথে অনেক বেশি ইউজার হ্যান্ডেল করতে পারবে।)*

### ৩. Nginx-এর প্রয়োজনীয়তা
ডেপ্লয়মেন্টে Nginx কেন জরুরি?
* **Reverse Proxy:** এটি ইউজার এবং আপনার পাইথন অ্যাপের মাঝখানে দেয়াল হিসেবে কাজ করে।
* **Static Files:** ইমেজ বা সিএসএস ফাইলগুলো পাইথন অ্যাপের কাছে না পাঠিয়ে Nginx নিজেই দ্রুত সার্ভ করতে পারে।
* **SSL (HTTPS):** আপনার সাইটকে নিরাপদ করার জন্য SSL সার্টিফিকেট Nginx-এ কনফিগার করা সহজ।
* **Load Balancing:** অনেক বেশি ট্রাফিক হলে Nginx একাধিক সার্ভার ইউনিটের মধ্যে রিকোয়েস্ট ভাগ করে দিতে পারে।

### ৫. ইন্টারভিউ টিপ:
যদি জিজ্ঞেস করে "সিঙ্গেল বনাম মাল্টি-কোর ডেপ্লয়মেন্ট কী?", বলবেন:
> *"ছোট অ্যাপে শুধু Uvicorn দিয়ে কাজ চালানো যায়, কিন্তু বড় এন্টারপ্রাইজ লেভেলে আমরা Gunicorn-এর সাথে Uvicorn ওয়ার্কার ব্যবহার করি যেন সার্ভারের সবকটি CPU কোর প্রপারলি ব্যবহার করা যায় এবং হাই অ্যাভেইল্যাবিলিটি (High Availability) নিশ্চিত হয়।"*

আপনার প্রিপারেশন এখন পূর্ণাঙ্গ। ডাটাবেস থেকে শুরু করে ডেপ্লয়মেন্ট—সব মেজর টপিক আমরা কভার করেছি। আপনার ইন্টারভিউয়ের জন্য অনেক শুভকামনা! সবশেষে কি আর কোনো ছোট বিষয় নিয়ে কথা বলব?

ইন্টারভিউয়ের শেষ পর্যায়ের জন্য এই তিনটি অপারেশনাল টপিক অত্যন্ত গুরুত্বপূর্ণ। নিচে সংক্ষেপে উত্তরগুলো দেওয়া হলো:

### ১. Health Check Endpoint
**এক কথায়:** এটি একটি সিম্পল এপিআই রাউট (সাধারণত `/health` বা `/status`), যা জানায় আপনার সার্ভিসটি বর্তমানে সচল (alive) আছে কি না।

* **কেন দরকার?**: লোড ব্যালেন্সার (Nginx) বা অর্কেস্ট্রেটর (Kubernetes) এটি চেক করে সিদ্ধান্ত নেয় যে ওই সার্ভারে রিকোয়েস্ট পাঠানো যাবে কি না।
* **Liveness vs Readiness**:
    * **Liveness:** সার্ভার কি চলছে? (শুধু `{"status": "ok"}` রিটার্ন করে)।
    * **Readiness:** সার্ভার কি কাজ করার জন্য প্রস্তুত? (এখানে ডাটাবেস বা ক্যাশ কানেকশন চেক করা হয়)।

```python
@app.get("/health")
async def health_check():
    # এখানে DB কানেকশন চেক করা ভালো
    return {"status": "healthy", "db": "connected"}
```

### ২. Logging in FastAPI
**এক কথায়:** অ্যাপ্লিকেশন চলার সময় কোথায় কী ঘটছে (Error, Warning, Info) তা ডায়েরির মতো লিখে রাখা।

* **Python `logging` module:** FastAPI সরাসরি পাইথনের স্ট্যান্ডার্ড `logging` মডিউল ব্যবহার করে।
* **কেন দরকার?**: প্রোডাকশনে কোনো এরর হলে ইউজার শুধু "Internal Server Error" দেখে, কিন্তু ডেভেলপার লগের মাধ্যমে আসল সমস্যা (Traceback) খুঁজে বের করে।
* **Structured Logging:** প্রোডাকশনে লগকে JSON ফরম্যাটে সেভ করা হয় যাতে **ELK Stack** বা **CloudWatch** দিয়ে সহজে সার্চ করা যায়।



---

### ৩. Monitoring
**এক কথায়:** আপনার অ্যাপ্লিকেশনের পারফরম্যান্স (CPU usage, RAM, Request per second, Error rate) রিয়েল-টাইমে পর্যবেক্ষণ করা।

**প্রধান ৩টি টুল যা ইন্টারভিউতে বলবেন:**
1.  **Prometheus:** এটি আপনার এপিআই থেকে ডাটা (Metrics) সংগ্রহ করে।
2.  **Grafana:** সংগ্রহ করা ডাটাগুলোকে সুন্দর গ্রাফ বা ড্যাশবোর্ড হিসেবে দেখায়।
3.  **Sentry:** এটি মূলত **Error Tracking**-এর জন্য। কোডে কোনো ক্র্যাশ বা এক্সেপশন হলে এটি সাথে সাথে ডেভেলপারকে নোটিফিকেশন পাঠায়।
