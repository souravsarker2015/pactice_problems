পাইথন এবং বিশেষ করে **Django** ফ্রেমওয়ার্ক ব্যবহার করে SQL ইনজেকশন এবং এর প্রতিকার নিচে সহজভাবে দেওয়া হলো:

### ১. SQL Injection কী?
SQL Injection হলো একটি সিকিউরিটি ভলনারেবিলিটি যেখানে হ্যাকার আপনার অ্যাপ্লিকেশনের ইনপুট ফিল্ডের (যেমন: লগইন ফর্ম বা ইউআরএল প্যারামিটার) মাধ্যমে ক্ষতিকর SQL কোড পাঠিয়ে দেয়। যদি আপনার কোড ইনপুটগুলোকে সরাসরি ডেটাবেস কুয়েরিতে বসিয়ে দেয়, তবে হ্যাকার আপনার পুরো ডেটাবেস মুছে ফেলতে পারে বা গোপন তথ্য চুরি করতে পারে।



---

### ২. Django কীভাবে এটি প্রতিরোধ করে?
Django ডিফল্টভাবেই SQL ইনজেকশন থেকে সুরক্ষা দেয়। এটি মূলত ৩টি উপায়ে কাজ করে:

#### ক) Django ORM-এর ব্যবহার
আপনি যখন `User.objects.filter(username=username)` ব্যবহার করেন, তখন Django সরাসরি কুয়েরি তৈরি না করে **Parameterization** ব্যবহার করে। অর্থাৎ, ইনপুটটি সরাসরি কোড হিসেবে রান না হয়ে একটি সাধারণ 'টেক্সট' বা ভ্যালু হিসেবে ডেটাবেসে যায়।

#### খ) Query Parameterization
ORM-এর আড়ালে Django আপনার ডেটা এবং SQL কুয়েরিকে আলাদা করে ফেলে। 
* **ভুল পদ্ধতি (Raw SQL with formatting):**
  ```python
  # এটি অত্যন্ত ঝুঁকিপূর্ণ!
  User.objects.raw(f"SELECT * FROM users WHERE name = '{user_input}'")
  ```
* **সঠিক পদ্ধতি (Using Params):**
  ```python
  # এটি নিরাপদ, কারণ Django ইনপুটকে কোটেশন দিয়ে স্যানিটাইজ করে নেয়
  User.objects.raw("SELECT * FROM users WHERE name = %s", [user_input])
  ```

#### গ) Template Auto-escaping
Django টেম্পলেট ইঞ্জিন অটোমেটিক্যালি সব ভেরিয়েবলকে এস্কেপ (Escape) করে। ফলে কোনো ইনপুটে যদি `<script>` বা SQL কমান্ড থাকে, তবে সেটি ব্রাউজারে বা ডেটাবেসে কোড হিসেবে এক্সিকিউট না হয়ে সাধারণ লেখা হিসেবে দেখায়।

---

### ৩. সেরা প্র্যাকটিস (Best Practices)

1.  **Avoid Raw SQL:** সবসময় Django-র বিল্ট-ইন **ORM** (filter, get, create) ব্যবহার করার চেষ্টা করুন। Raw SQL ব্যবহার করা থেকে বিরত থাকুন।
2.  **Never Use String Formatting:** কুয়েরির ভেতরে কখনো f-string বা `%` ব্যবহার করে ইউজার ইনপুট বসাবেন না।
3.  **Input Validation:** Django Forms বা Serializers ব্যবহার করুন। এগুলো ডেটাবেসে পাঠানোর আগেই ইনপুট ভ্যালিডেশন নিশ্চিত করে।
4.  **Least Privilege:** ডেটাবেস ইউজারকে শুধু প্রয়োজনীয় পারমিশন দিন (যেমন: শুধু `SELECT`, `INSERT`, `UPDATE`) যাতে ইনজেকশন হলেও সে পুরো ডেটাবেস ড্রপ করতে না পারে।

---

### এক নজরে সারসংক্ষেপ

| বৈশিষ্ট্য | ঝুঁকিপূর্ণ উপায় (Manual) | নিরাপদ উপায় (Django) |
| :--- | :--- | :--- |
| **কুয়েরি গঠন** | `"... WHERE id = " + user_id` | `filter(id=user_id)` |
| **পদ্ধতি** | String Concatenation | Query Parameterization |
| **ঝুঁকি** | হাই (SQL Injection সম্ভব) | জিরো (Django অটো স্যানিটাইজ করে) |

ইন্টারভিউতে বলবেন— *"Django-র ORM ব্যবহার করা হলে SQL ইনজেকশন হওয়ার সম্ভাবনা প্রায় থাকে না, কারণ এটি কুয়েরি এবং ডেটাকে আলাদাভাবে হ্যান্ডেল করে। তবে একান্তই Raw SQL ব্যবহার করতে হলে আমরা পাইথন স্ট্রিং ফরম্যাটিং না করে কুয়েরি প্যারামিটার ব্যবহার করি।"*

পরবর্তী টপিক হিসেবে **Cross-Site Scripting (XSS)** বা **CSRF Protection** নিয়ে আলোচনা করব?


**Cross-Site Scripting (XSS)** হলো একটি সিকিউরিটি ভলনারেবিলিটি যেখানে একজন হ্যাকার আপনার ওয়েবসাইটে ক্ষতিকর **JavaScript** কোড ইনজেক্ট করে দেয়। যখন কোনো সাধারণ ইউজার ওই পেজটি ভিজিট করেন, তখন সেই স্ক্রিপ্টটি ইউজারের ব্রাউজারে রান করে এবং হ্যাকার ইউজারের সেশন কুকি, টোকেন বা সেনসিটিভ তথ্য চুরি করতে পারে।

সহজ কথায়: SQL Injection আক্রমণ করে ডেটাবেসে, আর XSS আক্রমণ করে সরাসরি **ইউজারের ব্রাউজারে**।

---

### ১. XSS কত প্রকার?
ইন্টারভিউতে এই তিনটি টাইপ বলা খুব জরুরি:
* **Stored (Persistent) XSS:** ক্ষতিকর স্ক্রিপ্টটি পার্মানেন্টলি ডেটাবেসে সেভ হয়ে থাকে (যেমন: কমেন্ট সেকশন বা প্রোফাইল নেম)। যতজন ইউজার ওই পেজ দেখবেন, সবার ব্রাউজারে এটি রান করবে।
* **Reflected (Non-persistent) XSS:** স্ক্রিপ্টটি কোনো লিংকের প্যারামিটারে থাকে। ইউজার যখন ওই লিংকে ক্লিক করেন, তখন এটি রিফ্লেক্ট হয়ে ব্রাউজারে রান করে।
* **DOM-based XSS:** এটি পুরোপুরি ক্লায়েন্ট সাইডে ঘটে। জাভাস্ক্রিপ্ট কোড যখন অনিরাপদভাবে URL থেকে ডেটা নিয়ে সরাসরি পেজে রেন্ডার করে।



---

### ২. Django কীভাবে এটি প্রতিরোধ করে?
জ্যাঙ্গো ফ্রেমওয়ার্ক বাই-ডিফল্ট XSS থেকে অনেক শক্তিশালী সুরক্ষা দেয়:

#### ক) Template Auto-escaping
Django টেম্পলেট ইঞ্জিন সব ভেরিয়েবলকে অটোমেটিক্যালি **Escape** করে। 
* যদি কোনো ইউজার তার নাম হিসেবে `<script>alert('Hacked')</script>` ইনপুট দেয়, Django সেটাকে এভাবে রেন্ডার করবে: `&lt;script&gt;alert('Hacked')&lt;/script&gt;`।
* ব্রাউজার এটাকে কোড হিসেবে রান না করে সাধারণ টেক্সট হিসেবে স্ক্রিনে দেখাবে।

#### খ) `mark_safe` এবং `|safe` ফিল্টার
কখনো কখনো আমাদের HTML রেন্ডার করার প্রয়োজন হয়। তখন আমরা `|safe` ফিল্টার ব্যবহার করি। 
* **সতর্কতা:** শুধুমাত্র ট্রাস্টেড সোর্স (যেমন: অ্যাডমিন প্যানেল) থেকে আসা ডেটার ক্ষেত্রে এটি ব্যবহার করা উচিত। ইউজার ইনপুটে কখনোই এটি ব্যবহার করবেন না।

---

### ৩. সাধারণ প্রতিরোধ ব্যবস্থা (Best Practices)

1.  **Input Validation & Sanitization:** ইউজার যা ইনপুট দিচ্ছে তা সবসময় ভ্যালিডেট করুন। যেমন: ইমেইল ফিল্ডে যেন শুধু ইমেইলই থাকে।
2.  **Content Security Policy (CSP):** এটি একটি ব্রাউজার লেভেল সিকিউরিটি। এটি ঠিক করে দেয় আপনার ওয়েবসাইট কোন কোন সোর্স থেকে জাভাস্ক্রিপ্ট লোড করতে পারবে। এতে অজানা বা ক্ষতিকর স্ক্রিপ্ট রান হতে পারে না।
3.  **HTTP-Only Cookies:** সেশন কুকিগুলোকে `HttpOnly` হিসেবে সেট করুন। এতে জাভাস্ক্রিপ্ট (হ্যাকার) আপনার কুকি এক্সেস করতে পারবে না। Django-তে এটি ডিফল্টভাবেই থাকে: `SESSION_COOKIE_HTTPONLY = True`।
4.  **Use Modern Frameworks:** Django বা FastAPI-র মতো ফ্রেমওয়ার্ক ব্যবহার করুন যারা অটোমেটিক এস্কেপিং সাপোর্ট করে।

---

### ৪. ইন্টারভিউ সারসংক্ষেপ

| বৈশিষ্ট্য | SQL Injection | XSS |
| :--- | :--- | :--- |
| **টার্গেট** | সার্ভার/ডেটাবেস | ক্লায়েন্ট/ব্রাউজার |
| **ক্ষতি** | ডেটা চুরি বা ডিলিট হওয়া | সেশন হাইজ্যাকিং, কুকি চুরি |
| **প্রতিরোধ (Django)** | ORM/Parameterization | Auto-escaping/CSP |



ইন্টারভিউতে বলবেন— *"XSS প্রতিরোধের জন্য আমরা সবসময় Django-র ডিফল্ট টেম্পলেট ইঞ্জিন ব্যবহার করি যা অটোমেটিক এস্কেপিং নিশ্চিত করে। এছাড়া সেনসিটিভ কুকিগুলোকে `HttpOnly` করে রাখি এবং প্রোডাকশনে একটি শক্তিশালী **Content Security Policy (CSP)** ব্যবহার করি।"*

পরবর্তী টপিক হিসেবে **CSRF (Cross-Site Request Forgery)** নিয়ে আলোচনা করতে পারি? এটিও সিকিউরিটির জন্য অনেক গুরুত্বপূর্ণ।

**CSRF (Cross-Site Request Forgery)** হলো একটি সিকিউরিটি অ্যাটাক যেখানে একজন হ্যাকার একজন লগ-ইন থাকা ইউজারকে দিয়ে তার অজান্তেই কোনো ওয়েবসাইট বা ওয়েব অ্যাপ্লিকেশনে একটি ক্ষতিকর রিকোয়েস্ট (যেমন: পাসওয়ার্ড পরিবর্তন বা টাকা ট্রান্সফার) করিয়ে নেয়।

সহজ কথায়, হ্যাকার আপনার ব্রাউজারের "Trust" বা "Session" ব্যবহার করে আপনার হয়ে কাজ করে ফেলে।

---

### ১. CSRF কীভাবে কাজ করে?
ধরুন, আপনি আপনার ব্যাংকের সাইটে (bank.com) লগ-ইন করা আছেন। আপনার ব্রাউজারে সেই সাইটের একটি সেশন কুকি আছে। এখন আপনি যদি একই ব্রাউজারে অন্য একটি ট্যাবে হ্যাকারের কোনো লিংকে ক্লিক করেন, তবে সেই সাইট থেকে একটি ব্যাকগ্রাউন্ড রিকোয়েস্ট (POST request) bank.com-এ পাঠানো হতে পারে। যেহেতু আপনি অলরেডি লগ-ইন করা আছেন, ব্যাংক মনে করবে রিকোয়েস্টটি আপনিই পাঠিয়েছেন এবং সেটি কার্যকর করবে।



---

### ২. Django কীভাবে এটি প্রতিরোধ করে?
Django এই আক্রমণ ঠেকাতে **CSRF Token** সিস্টেম ব্যবহার করে। এটি মূলত একটি ইউনিক, সিক্রেট এবং র‍্যান্ডম ভ্যালু যা প্রতিবার পেজ রেন্ডার করার সময় জেনারেট হয়।

#### ক) CSRF Token (The Synchronizer Token Pattern)
যখনই আপনি কোনো ফর্মে ডাটা সাবমিট করেন, Django চেক করে যে সেই রিকোয়েস্টের সাথে সঠিক টোকেনটি আছে কি না। হ্যাকার বাইরের সাইট থেকে রিকোয়েস্ট পাঠাতে পারলেও, সে আপনার ইউনিক টোকেনটি চুরি করতে পারে না (Same-Origin Policy-র কারণে)। টোকেন না থাকলে Django ওই রিকোয়েস্ট রিজেক্ট করে দেয়।

#### খ) `{% csrf_token %}` ট্যাগ
Django টেম্পলেটে ফর্ম ব্যবহার করার সময় আমরা এই ট্যাগটি ব্যবহার করি:
```html
<form method="post">
    {% csrf_token %}
    <input type="text" name="amount">
    <button type="submit">Transfer</button>
</form>
```
এটি ব্যাকএন্ডে একটি হিডেন ইনপুট ফিল্ড তৈরি করে যেখানে টোকেনটি থাকে।

#### গ) CSRF Middleware
Django-র `CsrfViewMiddleware` অটোমেটিক্যালি সব ইনকামিং `POST`, `PUT`, `PATCH`, এবং `DELETE` রিকোয়েস্ট চেক করে। টোকেন না থাকলে এটি **403 Forbidden** এরর দেয়। তবে `GET` রিকোয়েস্টে এটি চেক করা হয় না, কারণ `GET` শুধু ডাটা দেখার জন্য, পরিবর্তনের জন্য নয়।

---

### ৩. API-এর ক্ষেত্রে CSRF (AJAX/Fetch)
যদি আপনি Django-র সাথে কোনো ফ্রন্টএন্ড ফ্রেমওয়ার্ক (যেমন: React, Vue) ব্যবহার করেন, তবে ডাটা পাঠানোর সময় আপনাকে রিকোয়েস্ট হেডারে **`X-CSRFToken`** পাঠাতে হবে। Django অটোমেটিক্যালি কুকি থেকে এই টোকেনটি রিড করতে পারে।

---

### ৪. ইন্টারভিউ টিপ: "Safe Methods"
ইন্টারভিউয়ার জিজ্ঞেস করতে পারেন— "কেন GET রিকোয়েস্টে CSRF টোকেন লাগে না?" 
উত্তর হবে: 
> "HTTP স্ট্যান্ডার্ড অনুযায়ী GET, HEAD, OPTIONS হলো 'Safe Methods'। এগুলো ডাটাবেসের কোনো স্টেট পরিবর্তন করে না। CSRF মূলত ডাটাবেসে ক্ষতিকর পরিবর্তন ঠেকানোর জন্য কাজ করে, তাই এটি শুধুমাত্র স্টেট-চেঞ্জিং মেথড (যেমন POST) এর জন্য জরুরি।"

---

### ৫. এক নজরে সারসংক্ষেপ

| অ্যাটাক | টার্গেট | জ্যাঙ্গোর সমাধান |
| :--- | :--- | :--- |
| **SQL Injection** | ডাটাবেস | ORM & Parameterization |
| **XSS** | ইউজার ব্রাউজার (Script) | Auto-escaping & CSP |
| **CSRF** | ইউজার সেশন (Action) | CSRF Token & Middleware |



পরবর্তী কোনো অ্যাডভান্সড সিকিউরিটি টপিক যেমন **CORS (Cross-Origin Resource Sharing)** নিয়ে কি আলোচনা করব? এটি মাইক্রোসার্ভিস বা ফ্রন্টএন্ড-ব্যাকএন্ড আলাদা হলে খুব গুরুত্বপূর্ণ।


**CORS (Cross-Origin Resource Sharing)** হলো একটি ব্রাউজার লেভেল সিকিউরিটি মেকানিজম যা নির্ধারণ করে একটি ডোমেইন থেকে অন্য ডোমেইনের রিসোর্স (যেমন API) কল করা যাবে কি না। এটি মূলত **Same-Origin Policy (SOP)** এর একটি আধুনিক এবং নিয়ন্ত্রিত শিথিলকরণ।

---

### ১. CORS কেন দরকার?
ব্রাউজার ডিফল্টভাবে এক ডোমেইন থেকে অন্য ডোমেইনের ডাটা এক্সেস করতে দেয় না (নিরাপত্তার স্বার্থে)। ধরুন, আপনার ফ্রন্টএন্ড চলছে `sourov.com`-এ এবং ব্যাকএন্ড চলছে `api.sourov.com`-এ। যেহেতু ডোমেইন আলাদা, ব্রাউজার সরাসরি কল ব্লক করে দেবে। এই ব্লক খোলার জন্যই CORS কনফিগার করতে হয়।



---

### ২. এটি কীভাবে কাজ করে? (Preflight Request)
যখন কোনো ব্রাউজার একটি "Complex" রিকোয়েস্ট (যেমন `POST` বা `PUT` যেখানে কাস্টম হেডার আছে) পাঠায়, তখন সে মূল রিকোয়েস্ট পাঠানোর আগে একটি **OPTIONS** রিকোয়েস্ট পাঠায়। একে বলা হয় **Preflight Request**। 
* ব্রাউজার জিজ্ঞেস করে: "আমি কি এই ডাটা পাঠাতে পারি?"
* সার্ভার যদি অনুমতি দেয় (Header-এ `Access-Control-Allow-Origin` পাঠিয়ে), তবেই ব্রাউজার মূল ডাটা পাঠায়।

---

### ৩. Django-তে CORS কনফিগারেশন
Django-তে এটি হ্যান্ডেল করার জন্য সবচেয়ে জনপ্রিয় লাইব্রেরি হলো `django-cors-headers`।

**ধাপগুলো:**
1.  ইনস্টল করা: `pip install django-cors-headers`
2.  `INSTALLED_APPS`-এ যুক্ত করা।
3.  `MIDDLEWARE`-এর একদম উপরে `CorsMiddleware` যুক্ত করা।
4.  **Settings**-এ অনুমতি দেওয়া:

```python
# নির্দিষ্ট কিছু ডোমেইন অ্যালাউ করতে:
CORS_ALLOWED_ORIGINS = [
    "http://localhost:3000",
    "https://sourov.com",
]

# অথবা সব ডোমেইন অ্যালাউ করতে (ডেভলপমেন্টের জন্য):
CORS_ALLOW_ALL_ORIGINS = True 
```

### ৪. গুরুত্বপূর্ণ হেডারসমূহ (Interview Points)
ইন্টারভিউতে এই হেডারগুলোর নাম বললে আপনার নলেজ গভীর মনে হবে:
* **Access-Control-Allow-Origin:** কোন ডোমেইন ডাটা পাবে।
* **Access-Control-Allow-Methods:** কোন মেথডগুলো (GET, POST, etc.) অ্যালাউড।
* **Access-Control-Allow-Headers:** কোন হেডারগুলো ক্লায়েন্ট পাঠাতে পারবে।
* **Access-Control-Allow-Credentials:** কুকি বা অথেন্টিকেশন টোকেন পাঠানোর অনুমতি দিবে কি না।

| **XSS** | Auto-escaping | User's Browser (Script) |

### ইন্টারভিউ টিপ:
যদি জিজ্ঞেস করে— *"প্রোডাকশনে কি `CORS_ALLOW_ALL_ORIGINS = True` রাখা উচিত?"*
উত্তর হবে— *"না, এটি একটি বড় সিকিউরিটি রিস্ক। প্রোডাকশনে শুধুমাত্র আমাদের ট্রাস্টেড ফ্রন্টএন্ড ডোমেইনগুলোকে `CORS_ALLOWED_ORIGINS` লিস্টে রাখা উচিত।"*

আপনার ইন্টারভিউয়ের জন্য পূর্ণাঙ্গ প্রস্তুতি হয়ে গেছে। পাইথন, জ্যাঙ্গো, ডাটাবেস এবং সিকিউরিটির বেসিক থেকে অ্যাডভান্সড সব টপিকই আমরা কভার করেছি। আপনার জন্য অনেক শুভকামনা! ইন্টারভিউ শেষে রেজাল্ট কেমন হলো জানাতে ভুলবেন না।

আপনি কি আর কোনো শেষ মুহূর্তের প্রশ্ন করতে চান?


ইন্টারভিউতে এই প্রশ্নটি খুবই কমন এবং উত্তর দেওয়ার সময় সহজ একটি উদাহরণ ব্যবহার করলে ইন্টারভিউয়ারের বুঝতে সুবিধা হয়। নিচে সংক্ষেপে পার্থক্যগুলো দেওয়া হলো:

### ১. মূল ধারণা (The Core Concept)

* **Authentication (অথেন্টিকেশন):** এটি হলো— **"আপনি কে?" (Who are you?)**। অর্থাৎ, ইউজারের পরিচয় নিশ্চিত করার প্রক্রিয়া। যেমন: ইউজারনেম, পাসওয়ার্ড বা ওটিপি (OTP) চেক করা।
* **Authorization (অথোরাইজেশন):** এটি হলো— **"আপনার কী কী করার অনুমতি আছে?" (What are you allowed to do?)**। অর্থাৎ, পরিচয় নিশ্চিত হওয়ার পর ইউজার কোন কোন ফাইল বা ফিচার অ্যাক্সেস করতে পারবেন তা নির্ধারণ করা।



---

### ২. প্রধান পার্থক্যসমূহ

| বৈশিষ্ট্য | Authentication | Authorization |
| :--- | :--- | :--- |
| **মূল প্রশ্ন** | আপনি কি সেই ব্যক্তি যার দাবি করছেন? | আপনি কি এই কাজটি করার অনুমতিপ্রাপ্ত? |
| **কখন ঘটে?** | এটি প্রথম ধাপ (লগইন করার সময়)। | এটি দ্বিতীয় ধাপ (লগইন হওয়ার পর)। |
| **কি চেক করা হয়?** | পাসওয়ার্ড, বায়োমেট্রিক্স, টোকেন। | ইউজার রোল (Admin, Manager, User), পারমিশন। |
| **ম্যানেজমেন্ট** | Identity Providers (যেমন: Google Login, Auth0)। | Access Control Lists (ACL), RBAC (Role Based Access Control)। |

---

### ৩. বাস্তব উদাহরণ (Real-life Example)
একটি অফিসের কথা চিন্তা করুন:
1.  **Authentication:** আপনি যখন গেটে আপনার আইডি কার্ড পাঞ্চ করে অফিসের ভেতরে ঢুকলেন, সেটা হলো অথেন্টিকেশন। অফিস নিশ্চিত হলো আপনি ওই অফিসেরই একজন কর্মচারী।
2.  **Authorization:** আপনি অফিসে ঢোকার পর আপনার নিজের ডেস্কে বসতে পারছেন, কিন্তু আপনি চাইলে বসের ব্যক্তিগত রুমে বা সার্ভার রুমে ঢুকতে পারছেন না। এই যে কার কোন রুমে ঢোকার পারমিশন আছে, সেটাই হলো অথোরাইজেশন।

---

### ৪. Django-তে এর প্রয়োগ
জ্যাঙ্গোতে এই দুটি বিষয় খুব চমৎকারভাবে হ্যান্ডেল করা হয়:
* **Authentication:** `authenticate(username=username, password=password)` ফাংশন ব্যবহার করে ইউজারের পাসওয়ার্ড চেক করা হয়।
* **Authorization:** `@permission_required` বা `@user_passes_test` ডেকোরেটর এবং জ্যাঙ্গোর **Groups** ও **Permissions** সিস্টেম ব্যবহার করে অথোরাইজেশন নিয়ন্ত্রণ করা হয়।



---

### ইন্টারভিউ টিপ:
ইন্টারভিউতে বলতে পারেন— *"অথেন্টিকেশন ছাড়া অথোরাইজেশন সম্ভব নয়। আগে ইউজারকে চিনতে হবে (Authentication), তারপর তাকে কাজ করার অনুমতি দিতে হবে (Authorization)।"*

আপনার ইন্টারভিউয়ের জন্য আর কোনো কনসেপ্ট (যেমন: **JWT**, **OAuth2**) কি বিস্তারিত জানার প্রয়োজন আছে? শুভকামনা!


**JWT (JSON Web Token)** হলো একটি ওপেন স্ট্যান্ডার্ড (RFC 7519) যা দুটি পক্ষের মধ্যে তথ্য আদান-প্রদানের একটি নিরাপদ এবং কম্প্যাক্ট উপায়। এটি মূলত **Stateless Authentication**-এর জন্য ব্যবহৃত হয়, অর্থাৎ সার্ভারকে প্রতিটি ইউজারের সেশন ডাটাবেসে বা মেমোরিতে সেভ করে রাখতে হয় না।

### ১. JWT কীভাবে কাজ করে? (How it works)

একটি JWT টোকেন মূলত তিনটি অংশ নিয়ে গঠিত: **Header**, **Payload**, এবং **Signature**। এগুলো ডট (.) দিয়ে আলাদা করা থাকে (যেমন: `xxxxx.yyyyy.zzzzz`)।



#### কাজের ধাপগুলো:
1.  **Authentication:** ইউজার তার ইউজারনেম এবং পাসওয়ার্ড দিয়ে লগইন করে।
2.  **Creation:** সার্ভার ইউজারের পরিচয় নিশ্চিত করে একটি JWT তৈরি করে। এই টোকেনে ইউজারের আইডি বা রোল (Payload) থাকে এবং এটি একটি সিক্রেট কি (Secret Key) দিয়ে সাইন করা হয়।
3.  **Storage:** সার্ভার টোকেনটি ক্লায়েন্টকে পাঠিয়ে দেয়। ক্লায়েন্ট সাধারণত এটি **LocalStorage** বা **Cookies**-এ সেভ করে।
4.  **Authorization:** পরবর্তী প্রতিটা রিকোয়েস্টে ক্লায়েন্ট হেডার হিসেবে টোকেনটি পাঠায়: `Authorization: Bearer <token>`।
5.  **Verification:** সার্ভার টোকেনটি পায় এবং নিজের সিক্রেট কি দিয়ে সিগনেচার চেক করে। যদি সিগনেচার মিলে যায়, তবে সার্ভার ইউজারকে অ্যাক্সেস দেয়। এখানে সার্ভারকে ডাটাবেস চেক করতে হয় না, কারণ সব তথ্য টোকেনের ভেতরেই থাকে।

---

### ২. JWT এর দুর্বলতাসমূহ (Weaknesses)

JWT খুব জনপ্রিয় হলেও এর কিছু সীমাবদ্ধতা ও ঝুঁকি রয়েছে:

* **টোকেন বাতিল করা কঠিন (Revocation Problem):** যেহেতু JWT স্টেটলেস, তাই একবার টোকেন ইস্যু করলে সেটি মেয়াদ শেষ না হওয়া পর্যন্ত কার্যকর থাকে। যদি কোনো ইউজারের অ্যাকাউন্ট হ্যাক হয় বা তাকে লগআউট করিয়ে দিতে চান, তবে সার্ভার সাইড থেকে সরাসরি সেই টোকেনটি "Kill" করা কঠিন (যদি না আপনি ব্ল্যাকলিস্টিং বা ডাটাবেস চেক যোগ করেন)।
* **সিক্রেট কি লিক হওয়া (Secret Key Vulnerability):** যদি আপনার সার্ভারের **Secret Key** কোনোভাবে হ্যাকার জেনে যায়, তবে সে যেকোনো ইউজারের জন্য ফেক টোকেন তৈরি করতে পারবে এবং আপনার পুরো সিস্টেমের নিয়ন্ত্রণ নিয়ে নেবে।
* **তথ্য দৃশ্যমানতা (Data Exposure):** JWT-র পে-লোড অংশটি শুধু **Base64** এনকোড করা থাকে, এনক্রিপ্ট করা নয়। তাই টোকেনটি যে কেউ ডিকোড করে ভেতরের তথ্য দেখতে পারে। একারণে JWT-র ভেতরে কখনো পাসওয়ার্ড বা সেনসিটিভ তথ্য রাখা উচিত নয়।
* **টোকেন চুরি (Token Theft/XSS):** যদি আপনি টোকেনটি `LocalStorage`-এ সেভ করেন, তবে **XSS (Cross-Site Scripting)** অ্যাটাকের মাধ্যমে হ্যাকার আপনার টোকেন চুরি করতে পারে।
* **Payload এর আকার (Size):** অনেক বেশি তথ্য পে-লোডে রাখলে টোকেনের সাইজ বড় হয়ে যায়, যা প্রতিবার নেটওয়ার্ক রিকোয়েস্টে বাড়তি ব্যান্ডউইথ খরচ করে।

---

### ৩. সুরক্ষা নিশ্চিত করার উপায় (Best Practices)

1.  **Short Expiry:** অ্যাক্সেস টোকেনের মেয়াদ খুব কম রাখা (যেমন ১৫ মিনিট) এবং দীর্ঘমেয়াদী ব্যবহারের জন্য **Refresh Token** ব্যবহার করা।
2.  **HttpOnly Cookies:** টোকেন স্টোর করার জন্য `LocalStorage`-এর চেয়ে `HttpOnly` এবং `Secure` কুকি বেশি নিরাপদ, যা XSS থেকে সুরক্ষা দেয়।
3.  **HTTPS:** টোকেন যেন মাঝপথ থেকে চুরি (Man-in-the-Middle attack) না হয়, সেজন্য সবসময় HTTPS ব্যবহার করা।
4.  **Sensitive Data:** পে-লোডে শুধুমাত্র প্রয়োজনীয় এবং নন-সেনসিটিভ ডাটা (যেমন: `user_id`, `role`) রাখা।



ইন্টারভিউতে বলবেন— *"JWT স্টেটলেস হওয়ার কারণে স্কেলেবিলিটির জন্য চমৎকার, কিন্তু এর বড় চ্যালেঞ্জ হলো টোকেন ইনভ্যালিডেশন। একারণে আমরা সাধারণত শর্ট-লাইভড টোকেন এবং রিফ্রেশ টোকেন প্যাটার্ন ব্যবহার করি।"*

ইন্টারভিউয়ের জন্য আপনি কি **Refresh Token** মেকানিজম বা **Token Blacklisting** সম্পর্কে আরও জানতে চান?


**OAuth2** (Open Authorization 2.0) হলো একটি স্ট্যান্ডার্ড **Authorization Protocol**। এটি কোনো ইউজারকে তার পাসওয়ার্ড শেয়ার না করেই একটি ওয়েবসাইট বা অ্যাপ্লিকেশনকে অন্য কোনো সার্ভিস (যেমন Google, Facebook বা GitHub) থেকে তার তথ্য ব্যবহারের অনুমতি দেওয়ার সুযোগ করে দেয়।

সহজ কথায়, এটি হলো একটি ডিজিটাল "কি-কার্ড" (Key-card) বা টোকেন সিস্টেম।

---

### ১. OAuth2 কীভাবে কাজ করে? (The Flow)

OAuth2-এর কাজ বোঝার জন্য চারটি প্রধান চরিত্রের কথা মাথায় রাখতে হবে:
1.  **Resource Owner:** আপনি নিজে (ইউজার)।
2.  **Client:** যে অ্যাপটি আপনি ব্যবহার করতে চান (যেমন: একটি নতুন গেম অ্যাপ)।
3.  **Authorization Server:** যে সার্ভারটি অনুমতি দেয় (যেমন: Google বা Facebook)।
4.  **Resource Server:** যেখানে আপনার আসল ডাটা রাখা আছে (যেমন: আপনার Google কন্টাক্টস বা প্রোফাইল)।



**কাজের ধাপসমূহ:**
* ইউজার অ্যাপে গিয়ে "Login with Google"-এ ক্লিক করে।
* অ্যাপটি ইউজারকে Google-এর পেজে পাঠায়।
* ইউজার Google-কে বলে, "হ্যাঁ, এই অ্যাপটিকে আমার নাম এবং ইমেইল দেখার অনুমতি দাও।"
* Google অ্যাপটিকে একটি **Authorization Code** দেয়।
* অ্যাপটি সেই কোডটি Google-কে ফেরত দিয়ে একটি **Access Token** সংগ্রহ করে।
* পরবর্তীতে ওই টোকেন দেখিয়ে অ্যাপটি ইউজারের তথ্য অ্যাক্সেস করে।

---

### ২. কখন আপনি OAuth2 ব্যবহার করবেন?

ইন্টারভিউতে এই পয়েন্টগুলো বলা খুব জরুরি। আপনি OAuth2 ব্যবহার করবেন যখন:

* **Third-party Integration:** আপনি যদি চান আপনার অ্যাপের ইউজাররা অন্য কোনো সার্ভিসের (যেমন: Google, LinkedIn) ডাটা আপনার অ্যাপে ব্যবহার করুক।
* **Single Sign-On (SSO):** যখন আপনি চান ইউজাররা আলাদা করে আপনার অ্যাপে অ্যাকাউন্ট না খুলে তাদের বিদ্যমান সামাজিক যোগাযোগ মাধ্যমের আইডি দিয়ে লগইন করুক।
* **Microservices Security:** বড় প্রজেক্টে যখন অনেকগুলো সার্ভিস থাকে, তখন একটি সেন্ট্রাল আইডেন্টিটি সার্ভার থেকে টোকেন ইস্যু করে সব সার্ভিসকে অথোরাইজ করার জন্য এটি সেরা উপায়।
* **Delegated Access:** আপনার পাসওয়ার্ড না দিয়ে অন্য কাউকে নির্দিষ্ট কিছু করার ক্ষমতা দিতে (যেমন: কোনো অ্যাপকে আপনার হয়ে টুইট করার অনুমতি দেওয়া)।

---

### ৩. OAuth2 বনাম JWT: কনফিউশন কাটান
ইন্টারভিউতে অনেকেই এই দুটোকে মিলিয়ে ফেলেন। মনে রাখবেন:
* **OAuth2** হলো একটি **প্রোটোকল** (ফ্রেমওয়ার্ক বা নিয়মাবলী)।
* **JWT** হলো একটি **টোকেন ফরম্যাট** যা প্রায়ই OAuth2 প্রোটোকলের ভেতরে তথ্য বহনের জন্য ব্যবহৃত হয়।

---

### ৪. OAuth2 এর প্রধান সুবিধা ও অসুবিধা

| সুবিধা | অসুবিধা |
| :--- | :--- |
| পাসওয়ার্ড শেয়ার করতে হয় না (High Security)। | ইমপ্লিমেন্ট করা বেশ জটিল। |
| পারমিশন লিমিট করা যায় (Scopes)। | ভুল কনফিগারেশনে সিকিউরিটি রিস্ক থাকে। |
| ইউজার এক্সপেরিয়েন্স ভালো হয় (দ্রুত লগইন)। | থার্ড-পার্টি সার্ভিসের ওপর নির্ভরতা বেড়ে যায়। |

---

### ইন্টারভিউ টিপ:
যদি জিজ্ঞেস করে— *"OAuth2 কি অথেন্টিকেশনের জন্য ব্যবহার করা যায়?"*
উত্তর হবে— *"টেকনিক্যালি OAuth2 হলো **Authorization** প্রোটোকল। তবে এর ওপর ভিত্তি করে তৈরি **OpenID Connect (OIDC)** লেয়ারটি **Authentication** হ্যান্ডেল করার জন্য ব্যবহৃত হয় (যেমন: Identity নিশ্চিত করা)।"*



আপনার ইন্টারভিউয়ের প্রস্তুতি এখন বেশ শক্তিশালী। প্রোটোকল, টোকেন, সিকিউরিটি এবং ডেপ্লয়মেন্ট—সব মেজর এরিয়া আপনি কভার করেছেন। আর কোনো নির্দিষ্ট টপিক কি বাকি আছে? শুভকামনা!


পাসওয়ার্ড সিকিউরলি স্টোর করা যেকোনো ব্যাকএন্ড ডেভেলপারের জন্য সবচেয়ে গুরুত্বপূর্ণ দায়িত্বগুলোর একটি। ইন্টারভিউতে এই প্রশ্নের উত্তরে নিচের ধাপগুলো ক্রমানুসারে বলা উচিত:

### ১. কখনোই প্লেইন টেক্সট সেভ করবেন না
সবচেয়ে মৌলিক নিয়ম হলো—পাসওয়ার্ড কখনোই সরাসরি বা প্লেইন টেক্সট হিসেবে ডাটাবেসে সেভ করা যাবে না। এমনকি ডাটাবেস অ্যাডমিনও যেন ইউজারের আসল পাসওয়ার্ড দেখতে না পারে।

### ২. পাসওয়ার্ড হ্যাশিং (Hashing)
পাসওয়ার্ডকে একটি **One-way Hash Function**-এর মাধ্যমে এনকোড করতে হবে। হ্যাশিং এমন একটি প্রক্রিয়া যা থেকে আসল পাসওয়ার্ডে ফিরে আসা (Decryption) অসম্ভব।

* **Hashing Algorithms:** বর্তমানে ইন্টারভিউতে **Argon2** অথবা **bcrypt**-এর কথা বলা সবচেয়ে নিরাপদ। MD5 বা SHA1 এখন অত্যন্ত দুর্বল এবং এগুলো ব্যবহার করা নিষিদ্ধ।



### ৩. সল্টিং (Salting)
শুধুমাত্র হ্যাশিং যথেষ্ট নয়, কারণ হ্যাকাররা **Rainbow Tables** (আগে থেকে জেনারেট করা হ্যাশ লিস্ট) ব্যবহার করে পাসওয়ার্ড বের করে ফেলতে পারে।
* **Salt:** এটি একটি র‍্যান্ডম স্ট্রিং যা হ্যাশ করার আগে পাসওয়ার্ডের সাথে যোগ করা হয়। 
* **কেন দরকার?**: যদি দুইজন ইউজারের পাসওয়ার্ড একই হয় (যেমন: `123456`), তবে সল্টিং-এর কারণে তাদের হ্যাশ ভ্যালু ডাটাবেসে সম্পূর্ণ আলাদা দেখাবে।



### ৪. কেন ডিক্রিপশন (Decryption) নয়?
ইন্টারভিউয়ার ট্রিক করার জন্য জিজ্ঞেস করতে পারেন— "কেন পাসওয়ার্ড এনক্রিপ্ট না করে হ্যাশ করি?"
* **উত্তর:** এনক্রিপশন হলো 'Two-way', অর্থাৎ কি (Key) থাকলে তা ডিক্রিপ্ট করে আসল পাসওয়ার্ড দেখা সম্ভব। কিন্তু আমাদের পাসওয়ার্ড দেখার কোনো দরকার নেই, শুধু 'Match' করা দরকার। তাই হ্যাশিং নিরাপদ।

---

### ৫. জ্যাঙ্গো কীভাবে এটি হ্যান্ডেল করে?
জ্যাঙ্গোতে পাসওয়ার্ড স্টোর করার জন্য আপনাকে ম্যানুয়ালি কিছু করতে হয় না। জ্যাঙ্গো ডিফল্টভাবে **PBKDF2** অ্যালগরিদম এবং **SHA256** হ্যাশ ব্যবহার করে যা অত্যন্ত নিরাপদ।
* যখন আপনি `user.set_password('password123')` কল করেন, জ্যাঙ্গো অটোমেটিক্যালি সল্ট জেনারেট করে এবং হ্যাশ করে সেভ করে।

---

### ৬. সেরা প্র্যাকটিস (Checklist for Interview)

| বৈশিষ্ট্য | কেন করবেন? |
| :--- | :--- |
| **Strong Algorithm** | **Argon2**, **bcrypt**, বা **scrypt** ব্যবহার করা। |
| **Unique Salt** | প্রতিটা ইউজারের জন্য আলাদা র‍্যান্ডম সল্ট ব্যবহার করা। |
| **Work Factor** | হ্যাশিং প্রসেসকে ইচ্ছাকৃতভাবে কিছুটা স্লো করা (Cost factor), যাতে ব্রুট ফোর্স অ্যাটাক কঠিন হয়। |
| **HTTPS** | ডাটাবেসে সেভ করার আগে যখন ইউজার পাসওয়ার্ড পাঠায়, সেই ট্রানজিট যেন নিরাপদ থাকে। |

### ইন্টারভিউ ক্লোজিং স্টেটমেন্ট:
> *"পাসওয়ার্ড সিকিউরলি স্টোর করার জন্য আমরা আধুনিক হ্যাশিং অ্যালগরিদম যেমন **bcrypt** ব্যবহার করি এবং সাথে একটি ইউনিক **Salt** যোগ করি। এছাড়া ডাটাবেস লেভেলের সুরক্ষার পাশাপাশি আমরা ইনকামিং রিকোয়েস্টের জন্য **HTTPS** নিশ্চিত করি যেন মাঝপথ থেকে পাসওয়ার্ড চুরি না হয়।"*

আপনার প্রিপারেশন এখন ১০০% কমপ্লিট। আশা করছি আপনি চমৎকার একটি ইন্টারভিউ দেবেন। শুভকামনা!


**HTTPS (Hypertext Transfer Protocol Secure)** হলো ইন্টারনেটে ডাটা আদান-প্রদান করার একটি নিরাপদ পদ্ধতি। এটি সাধারণ HTTP-র একটি উন্নত সংস্করণ, যেখানে নিরাপত্তা নিশ্চিত করার জন্য **SSL/TLS** এনক্রিপশন লেয়ার যোগ করা হয়।

সহজ কথায়, আপনি যখন কোনো ওয়েবসাইটে তথ্য (যেমন পাসওয়ার্ড বা ক্রেডিট কার্ড নম্বর) টাইপ করেন, HTTPS সেই তথ্যগুলোকে এমনভাবে এনকোড করে দেয় যাতে মাঝপথে কেউ সেগুলো চুরি করলেও পড়তে না পারে।

---

### ১. HTTPS কীভাবে কাজ করে?
HTTPS মূলত তিনটি স্তরে নিরাপত্তা প্রদান করে:
1.  **Encryption:** আদান-প্রদান করা ডাটা এনক্রিপ্ট করা হয় যাতে হ্যাকাররা তা বুঝতে না পারে।
2.  **Data Integrity:** ডাটা ট্রান্সফার করার সময় কেউ তাতে কোনো পরিবর্তন বা কাটছাঁট করতে পারে না।
3.  **Authentication:** এটি নিশ্চিত করে যে ইউজার সঠিক ওয়েবসাইটের সাথেই কথা বলছেন (অর্থাৎ, আপনি bank.com এ গেলে সেটি আসল ব্যাংক কি না তা নিশ্চিত করে)।



---

### ২. এটি কেন গুরুত্বপূর্ণ? (Why it matters)

ইন্টারভিউতে গুরুত্ব বোঝাতে নিচের ৪টি পয়েন্ট বলতে পারেন:

* **Security (নিরাপত্তা):** ইউজার এবং সার্ভারের মধ্যে গোপনীয়তা বজায় থাকে। বিশেষ করে পাবলিক ওয়াইফাই (যেমন ক্যাফে বা এয়ারপোর্ট) ব্যবহার করার সময় HTTPS না থাকলে ডাটা চুরি হওয়ার ঝুঁকি অনেক বেশি।
* **Trust & SEO:** গুগল বর্তমানে HTTPS ছাড়া সাইটগুলোকে "Not Secure" হিসেবে দেখায়। এছাড়া, যে সাইটগুলোতে HTTPS থাকে, গুগল সার্চে সেগুলোকে র‍্যাঙ্কিংয়ে অগ্রাধিকার দেয়।
* **Preventing Attacks:** এটি **Man-in-the-Middle (MitM)** অ্যাটাক প্রতিরোধ করে। যেখানে হ্যাকার নিজেকে ইউজার এবং সার্ভারের মাঝখানে বসিয়ে সব তথ্য দেখে ফেলে।
* **Legal Compliance:** আর্থিক লেনদেন বা ব্যক্তিগত তথ্য সংগ্রহের জন্য বর্তমানে প্রায় সব দেশেই HTTPS ব্যবহার করা আইনত বাধ্যতামূলক (যেমন: PCI-DSS বা GDPR কমপ্লায়েন্স)।

---

### ৩. SSL Handshake প্রক্রিয়া
ইন্টারভিউয়ার যদি একটু গভীরে যেতে চান, তবে **SSL Handshake** সম্পর্কে বলতে পারেন। এটি হলো সেই প্রক্রিয়া যার মাধ্যমে ব্রাউজার এবং সার্ভার একে অপরের সাথে পরিচিত হয় এবং এনক্রিপশন কি (Key) শেয়ার করে।

| বৈশিষ্ট্য | HTTP | HTTPS |
| :--- | :--- | :--- |
| **সিকিউরিটি** | অনিরাপদ (প্লেইন টেক্সট)। | নিরাপদ (এনক্রিপ্টেড)। |
| **পোর্ট** | ডিফল্ট পোর্ট **80**। | ডিফল্ট পোর্ট **443**। |
| **সার্টিফিকেট** | কোনো সার্টিফিকেটের প্রয়োজন নেই। | **SSL/TLS Certificate** প্রয়োজন। |
| **ব্রাউজার স্ট্যাটাস** | "Not Secure" দেখায়। | "Padlock" আইকন দেখায়। |

### ইন্টারভিউ টিপ:
ইন্টারভিউতে বলবেন— *"ডেভেলপার হিসেবে আমরা যখন কোনো এপিআই (FastAPI বা Django) ডেপ্লয় করি, তখন আমরা এনজিনেক্স (Nginx) বা ক্লাউডফ্লেয়ার (Cloudflare) ব্যবহার করে HTTPS নিশ্চিত করি। কারণ এর ফলে ট্রানজিটে থাকা সমস্ত ডাটা (যেমন JWT টোকেন বা ইউজার পাসওয়ার্ড) সুরক্ষিত থাকে।"*

আপনার প্রিপারেশন এখন অত্যন্ত মজবুত। শুভকামনা আপনার ইন্টারভিউয়ের জন্য! আপনি কি আর কোনো নেটওয়ার্কিং বা সিকিউরিটি টপিক নিয়ে আলোচনা করতে চান?


**Security Headers** হলো HTTP রেসপন্স হেডারের একটি সেট যা ব্রাউজারকে নির্দেশ দেয় কীভাবে আপনার সাইটের সাথে আচরণ করতে হবে এবং কোন কোন সিকিউরিটি পলিসি মেনে চলতে হবে। এটি মূলত ব্রাউজার লেভেলে একটি শক্তিশালী নিরাপত্তা দেয়াল তৈরি করে।

ইন্টারভিউতে একটি প্রোডাকশন-রেডি API-এর জন্য নিচের হেডারগুলোর কথা বলা সবচেয়ে গুরুত্বপূর্ণ:

### ১. প্রধান সিকিউরিটি হেডারসমূহ

| হেডার (Header) | কেন ব্যবহার করবেন? (Purpose) |
| :--- | :--- |
| **Strict-Transport-Security (HSTS)** | এটি ব্রাউজারকে বাধ্য করে যেন সব কমিউনিকেশন শুধুমাত্র **HTTPS**-এ হয়। এটি 'Protocol Downgrade' অ্যাটাক ঠেকায়। |
| **Content-Security-Policy (CSP)** | এটি নির্ধারণ করে দেয় কোন কোন সোর্স থেকে স্ক্রিপ্ট বা ইমেজ লোড হতে পারবে। এটি **XSS** ঠেকানোর সবচেয়ে শক্তিশালী উপায়। |
| **X-Frame-Options** | এটি আপনার সাইটকে অন্য কোনো সাইটের ভেতর `<iframe>` হিসেবে লোড হতে বাধা দেয়। এটি **Clickjacking** অ্যাটাক প্রতিরোধ করে। |
| **X-Content-Type-Options** | এটি ব্রাউজারকে ফাইল টাইপ নিজে থেকে গেস (MIME sniffing) করতে নিষেধ করে। ফলে হ্যাকাররা ইমেজ ফাইলের আড়ালে স্ক্রিপ্ট চালাতে পারে না। |
| **Referrer-Policy** | ইউজার যখন আপনার সাইট থেকে অন্য সাইটে যায়, তখন আপনার সাইটের কতটুকু তথ্য (URL) পাঠানো হবে তা নিয়ন্ত্রণ করে। |
| **Permissions-Policy** | এটি ব্রাউজারের বিভিন্ন ফিচার যেমন ক্যামেরা, মাইক্রোফোন বা জিওলোকেশন কোন পরিস্থিতিতে ব্যবহার করা যাবে তা নিয়ন্ত্রণ করে। |

---

### ২. Django-তে এই হেডারগুলো কীভাবে যুক্ত করবেন?

Django-তে এই হেডারগুলো যুক্ত করা খুবই সহজ কারণ এর বিল্ট-ইন `SecurityMiddleware` এগুলো হ্যান্ডেল করে। আপনাকে শুধু `settings.py`-তে কিছু কনফিগারেশন করতে হবে:

```python
# settings.py

# HSTS কনফিগারেশন (প্রোডাকশনের জন্য)
SECURE_HSTS_SECONDS = 31536000  # ১ বছর
SECURE_HSTS_INCLUDE_SUBDOMAINS = True
SECURE_HSTS_PRELOAD = True

# HTTPS রিডাইরেক্ট
SECURE_SSL_REDIRECT = True

# X-Frame-Options
X_FRAME_OPTIONS = 'DENY'

# X-Content-Type-Options
SECURE_CONTENT_TYPE_NOSNIFF = True

# Browser XSS Filter (পুরানো ব্রাউজারের জন্য)
SECURE_BROWSER_XSS_FILTER = True
```

### ৩. ইন্টারভিউ টিপ: "Insecure Headers" মুছে ফেলা
ইন্টারভিউতে একটি প্রো টিপ দিতে পারেন— আমরা শুধু হেডার যোগই করি না, কিছু হেডার **মুছেও ফেলি**। 
* যেমন: **`Server`** বা **`X-Powered-By`** হেডারগুলো রিমুভ করা উচিত। 
* কারণ এগুলো হ্যাকারকে জানিয়ে দেয় আপনি কোন সার্ভার (যেমন Nginx/Apache) বা কোন ফ্রেমওয়ার্ক (Django/FastAPI) ব্যবহার করছেন। তথ্য গোপন রাখা (Security by Obscurity) এখানে একটি ভালো প্র্যাকটিস।

### ৪. কিভাবে চেক করবেন?
আপনি ইন্টারভিউতে বলতে পারেন যে, আমি সাধারণত **[SecurityHeaders.com](https://securityheaders.com)** এর মতো টুল ব্যবহার করি চেক করার জন্য যে আমার API-তে সব প্রয়োজনীয় হেডার সঠিকভাবে সেট করা আছে কি না।

**সারসংক্ষেপ:**
Security headers সরাসরি আপনার কোডের বাগ ফিক্স না করলেও, এটি ব্রাউজারকে একটি হার্ডেনড (Hardened) বা শক্তিশালী সুরক্ষা কবচ দেয় যা অনেক কমন ওয়েব অ্যাটাক থেকে আপনার সিস্টেমকে রক্ষা করে।

আপনার ইন্টারভিউয়ের প্রস্তুতি এখন একদম টপ-নচ! আর কোনো কনফিউশন আছে কি?

লগইন এপিআই-তে **Brute Force Attack** (বারবার ভুল পাসওয়ার্ড দিয়ে লগইন করার চেষ্টা) প্রতিরোধ করা একটি অত্যন্ত গুরুত্বপূর্ণ সিকিউরিটি স্টেপ। ইন্টারভিউতে এই প্রশ্নের উত্তরে আপনি নিচের ৫টি কৌশলের কথা বলতে পারেন:

### ১. Rate Limiting (রেট লিমিটিং)
এটি সবচেয়ে কার্যকর পদ্ধতি। এর মাধ্যমে আপনি নির্দিষ্ট সময়ের মধ্যে (যেমন ১ মিনিটে ৫ বার) একজন ইউজার বা একটি IP এড্রেস কতবার রিকোয়েস্ট পাঠাতে পারবে তা সীমাবদ্ধ করে দেন।
* **Django:** আপনি `django-ratelimit` লাইব্রেরি ব্যবহার করতে পারেন।
* **FastAPI:** `slowapi` বা সরাসরি Nginx লেভেলে রেট লিমিটিং সেট করা যায়।

### ২. Account Lockout (অ্যাকাউন্ট লকআউট)
যদি কেউ পর পর কয়েকবার (যেমন ৫ বার) ভুল পাসওয়ার্ড দেয়, তবে তার অ্যাকাউন্টটি সাময়িকভাবে (যেমন ৩০ মিনিটের জন্য) লক করে দেওয়া।
* জ্যাঙ্গোতে এটি করার জন্য `django-axes` নামক একটি দারুণ লাইব্রেরি আছে যা অটোমেটিক্যালি ব্যর্থ লগইন চেষ্টাগুলো ট্র্যাক করে এবং নির্দিষ্ট সীমার পর অ্যাক্সেস ব্লক করে দেয়।

### ৩. Multi-Factor Authentication (MFA/2FA)
শুধুমাত্র পাসওয়ার্ডের ওপর নির্ভর না করে অতিরিক্ত একটি ভেরিফিকেশন লেয়ার যোগ করা (যেমন ইমেইল ওটিপি বা Google Authenticator)। এতে হ্যাকার পাসওয়ার্ড জানলেও লগইন করতে পারবে না।



### ৪. CAPTCHA ইমপ্লিমেন্টেশন
যখন সিস্টেম দেখবে যে একই IP থেকে অনেক বেশি রিকোয়েস্ট আসছে, তখন আপনি **Google reCAPTCHA** চ্যালেঞ্জ ছুড়ে দিতে পারেন। এটি নিশ্চিত করে যে রিকোয়েস্টটি মানুষ পাঠাচ্ছে, কোনো অটোমেটেড বট নয়।

### ৫. Progressive Delays (Throttling)
এটি একটি স্মার্ট পদ্ধতি। প্রতিবার ভুল পাসওয়ার্ড দেওয়ার পর পরবর্তী চেষ্টার জন্য ওয়েটিং টাইম বাড়িয়ে দেওয়া। যেমন: প্রথমবার ভুল হলে ২ সেকেন্ড অপেক্ষা, দ্বিতীয়বার ৫ সেকেন্ড, তৃতীয়বার ১০ সেকেন্ড। এটি বটের জন্য অ্যাটাক করা অনেক স্লো এবং বিরক্তিকর করে তোলে।

---

### ৬. টেকনিক্যাল চেকলিস্ট (Best Practices)

| পদ্ধতি | কাজ করার ধরন |
| :--- | :--- |
| **IP-based Blocking** | সন্দেহজনক IP এড্রেসগুলোকে সাময়িকভাবে ব্লক করা। |
| **Email Notification** | একাধিকবার ব্যর্থ চেষ্টার পর ইউজারকে ইমেইল পাঠিয়ে জানানো। |
| **Monitoring logs** | `fail2ban` এর মতো টুল দিয়ে সার্ভার লগ মনিটর করা এবং অ্যাটাকারকে ব্লক করা। |
| **Complex Passwords** | ইউজারকে শক্তিশালী পাসওয়ার্ড সেট করতে বাধ্য করা। |

### ৭. ইন্টারভিউ টিপ:
ইন্টারভিউয়ার যদি জিজ্ঞেস করে "IP-based blocking কি যথেষ্ট?", আপনি বলবেন:
> *"না, কারণ হ্যাকাররা প্রক্সি বা ভিপিএন ব্যবহার করে IP পরিবর্তন করতে পারে। তাই আমাদের IP-based লিমিটিংয়ের পাশাপাশি **Username-based blocking** এবং **MFA** ব্যবহার করা উচিত যেন সিকিউরিটি লেয়ারটি আরও মজবুত হয়।"*



আপনার সব প্রিপারেশন এখন একদম সলিড। ইন্টারভিউতে আত্মবিশ্বাসের সাথে উত্তর দিন। আর কোনো প্রশ্ন আছে? শুভকামনা!


**Principle of Least Privilege (PoLP)** হলো সাইবার সিকিউরিটির একটি মৌলিক ধারণা। এর মূল কথা হলো—একজন ইউজার, প্রোগ্রাম বা সিস্টেমকে শুধুমাত্র সেইটুকু অ্যাক্সেস বা পারমিশন দেওয়া উচিত যা তার কাজ সম্পন্ন করার জন্য **ন্যূনতম প্রয়োজন**।

সহজ কথায়, যার যতটুকু দরকার, তাকে ঠিক ততটুকুই পাওয়ার দেওয়া; তার বেশি এক বিন্দুও নয়।

---

### ১. একটি বাস্তব উদাহরণ
ধরুন, আপনার কোম্পানিতে একজন **Content Writer** আছেন। 
* তাকে শুধু ব্লগ পোস্ট লেখা এবং এডিট করার পারমিশন দেওয়া উচিত। 
* তাকে যদি ভুলে পুরো ওয়েবসাইটের 'Admin' বা 'Database Delete' করার ক্ষমতা দেওয়া হয়, তবে সেটি হবে PoLP-এর লঙ্ঘন। কারণ তার কাজ করার জন্য ডাটাবেস ডিলিট করার কোনো প্রয়োজন নেই।



---

### ২. এটি কেন গুরুত্বপূর্ণ? (Why it matters)

ইন্টারভিউতে এর গুরুত্ব বোঝাতে নিচের ৩টি পয়েন্ট বলতে পারেন:

* **Attacking Surface কমানো:** যদি কোনো ইউজারের অ্যাকাউন্ট হ্যাক হয়, তবে হ্যাকার শুধুমাত্র ওই ইউজারের সীমিত পারমিশনগুলোই পাবে। ইউজারের ক্ষমতা কম থাকলে হ্যাকারের পক্ষে পুরো সিস্টেমের ক্ষতি করা কঠিন হয়ে পড়ে।
* **ভুলবশত ক্ষতি রোধ (Accidental Damage):** অনেক সময় ডেভেলপার বা সিস্টেম অ্যাডমিনরা ভুল কমান্ড দিয়ে ডাটা মুছে ফেলতে পারেন। PoLP থাকলে তাদের কাছে সব সময় সব ডিলিট করার ক্ষমতা থাকে না, যা বড় দুর্ঘটনা থেকে বাঁচায়।
* **ইনসাইডার থ্রেট (Insider Threats):** কোনো অসৎ কর্মচারী যেন তার পদের বাইরে গিয়ে কোম্পানির গোপন তথ্য চুরি করতে না পারে, তা নিশ্চিত করে এই নীতি।

---

### ৩. টেকনিক্যাল প্রয়োগ (In Development & DevOps)

ডেভেলপার হিসেবে আপনি এটি যেভাবে ইমপ্লিমেন্ট করবেন:

1.  **Database Access:** আপনার অ্যাপ্লিকেশনকে ডাটাবেসের `root` বা `superuser` এক্সেস দেবেন না। পরিবর্তে এমন একটি ইউজার তৈরি করুন যে শুধু নির্দিষ্ট টেবিলগুলোতে `SELECT`, `INSERT`, বা `UPDATE` করতে পারে।
2.  **Server Access (SSH):** সবাইকে সার্ভারের `sudo` বা `root` এক্সেস না দিয়ে শুধু প্রয়োজনীয় ফোল্ডারে কাজ করার অনুমতি দেওয়া।
3.  **Role-Based Access Control (RBAC):** জ্যাঙ্গোতে যেমন আমরা `Admin`, `Staff`, এবং `Customer` রোল আলাদা করি—এটিও PoLP-এর একটি অংশ।
4.  **Microservices:** একটি সার্ভিস যেন অন্য সার্ভিসের শুধুমাত্র প্রয়োজনীয় API এন্ডপয়েন্টগুলোই কল করতে পারে।

### ৪. ইন্টারভিউ সারসংক্ষেপ

| বৈশিষ্ট্য | সাধারণ পদ্ধতি (Bad) | PoLP পদ্ধতি (Good) |
| :--- | :--- | :--- |
| **অ্যাক্সেস লেভেল** | সবাইকে সবকিছুর ফুল এক্সেস দেওয়া। | শুধুমাত্র কাজের জন্য যতটুকু দরকার। |
| **ঝুঁকি** | অনেক বেশি (এক জন হ্যাক হলে সব শেষ)। | অনেক কম (ক্ষতি নির্দিষ্ট সীমার মধ্যে থাকে)। |
| **ম্যানেজমেন্ট** | সহজ, কিন্তু অনিরাপদ। | একটু সময়সাপেক্ষ, কিন্তু অত্যন্ত নিরাপদ। |

### ৫. ইন্টারভিউ টিপ:
যদি জিজ্ঞেস করে "কিভাবে এটি শুরু করব?", বলবেন:
> *"আমরা ডিফল্টভাবে সবার এক্সেস **Deny** (বন্ধ) করে রাখি। এরপর যার যেটুকু পারমিশন লাগে, তা ধীরে ধীরে যোগ করি। একে বলা হয় **Default Deny** এপ্রোচ।"*

আপনার সিকিউরিটি রিলেটেড নলেজ এখন বেশ গভীর। এটিই ছিল আমাদের আলোচনার শেষ মেজর সিকিউরিটি টপিক। ইন্টারভিউতে আপনি এখন যেকোনো আর্কিটেকচারাল বা সিকিউরিটি প্রশ্নে খুব আত্মবিশ্বাসের সাথে উত্তর দিতে পারবেন।

আপনার ইন্টারভিউ কবে? আর কোনো বিষয়ে কি সাহায্য করতে পারি?

কোডবেসে সেনসিটিভ ডাটা (যেমন: পাসওয়ার্ড, ক্রেডিট কার্ড নম্বর বা এপিআই কি) হ্যান্ডেল করা অত্যন্ত স্পর্শকাতর বিষয়। ইন্টারভিউতে এই প্রশ্নের উত্তরে আপনাকে প্রমাণ করতে হবে যে আপনি সিকিউরিটির ব্যাপারে খুবই সচেতন।

নিচে ধাপে ধাপে সঠিক পদ্ধতিগুলো দেওয়া হলো:

### ১. পাসওয়ার্ড এবং সিক্রেট কি (Environment Variables)
কোডের ভেতরে সরাসরি (Hardcoded) কখনো কোনো পাসওয়ার্ড বা সিক্রেট কি রাখা যাবে না। 
* **কেন?**: যদি আপনি কোডটি GitHub-এ পুশ করেন, তবে যে কেউ আপনার ডাটাবেস বা থার্ড-পার্টি সার্ভিসের এক্সেস পেয়ে যাবে।
* **সমাধান:** সবসময় **Environment Variables (`.env` file)** ব্যবহার করতে হবে। জ্যাঙ্গোতে আমরা `python-dotenv` বা `django-environ` লাইব্রেরি ব্যবহার করি।

> **প্রো-টিপ:** সবসময় নিশ্চিত করতে হবে যেন `.gitignore` ফাইলে `.env` ফাইলটি যুক্ত থাকে।

---

### ২. ক্রেডিট কার্ড ডাটা (PCI-DSS Compliance)
ক্রেডিট কার্ড নম্বর বা CVV কখনোই নিজের ডাটাবেসে সেভ করা উচিত নয়।
* **কেন?**: কার্ডের তথ্য সেভ করার জন্য অনেক কঠোর আইনি নিয়ম (PCI-DSS) মানতে হয়। ডাটা লিক হলে আপনার কোম্পানি বড় আইনি ঝামেলায় পড়বে।
* **সমাধান:** **Payment Gateway** (যেমন: Stripe, SSLCommerz, বা PayPal) ব্যবহার করুন। তারা আপনাকে একটি **Token** দেয়। আপনি শুধু সেই টোকেনটি সেভ করবেন, আসল কার্ড নম্বর নয়।



---

### ৩. পাসওয়ার্ড স্টোরেজ (Hashing)
পাসওয়ার্ড কখনোই প্লেইন টেক্সট হিসেবে সেভ করা যাবে না। 
* **সমাধান:** সবসময় একটি শক্তিশালী **One-way Hashing Algorithm** (যেমন: `bcrypt` বা `Argon2`) এবং **Salt** ব্যবহার করতে হবে। জ্যাঙ্গো এটি ডিফল্টভাবেই হ্যান্ডেল করে।

---

### ৪. লগিং এবং মাস্কিং (Logging & Masking)
অ্যাপ্লিকেশনের এরর বা ডিবাগিং লগে যেন ভুলবশত সেনসিটিভ ডাটা চলে না যায় তা নিশ্চিত করতে হয়।
* **Masking:** লগ ফাইলে ডাটা দেখানোর সময় `****` ব্যবহার করা। যেমন: `Card: 4242 **** **** 1234`।
* **Sensitive Parameters:** জ্যাঙ্গোতে `@sensitive_post_parameters()` ডেকোরেটর ব্যবহার করা যায় যা নির্দিষ্ট প্যারামিটারগুলোকে লগে আসা থেকে বিরত রাখে।

---

### ৫. ডাটাবেস লেভেল সিকিউরিটি (Encryption at Rest)
যদি কোনো বিশেষ প্রয়োজনে সেনসিটিভ ডাটা (যেমন: ন্যাশনাল আইডি নম্বর) সেভ করতেই হয়, তবে তা সরাসরি না করে **Encryption at Rest** পদ্ধতি ব্যবহার করতে হবে। অর্থাৎ ডাটাবেসে সেভ হওয়ার আগেই ডাটাটি এনক্রিপ্ট হয়ে যাবে এবং পড়ার সময় ডিক্রিপ্ট হবে।

### ৬. ইন্টারভিউ সারসংক্ষেপ (Best Practices Checklist)

| ক্যাটাগরি | করণীয় |
| :--- | :--- |
| **API Keys/Secrets** | ব্যবহার করুন `.env` ফাইল এবং এনভায়রনমেন্ট ভ্যারিয়েবল। |
| **Passwords** | সল্টিং এবং হ্যাশিং (bcrypt) ব্যবহার করুন। |
| **Card Details** | টোকেনাইজেশন ব্যবহার করুন (Stripe/Payment Gateway)। |
| **Data in Transit** | সবসময় **HTTPS/TLS** ব্যবহার করুন। |
| **Git Safety** | কখনো `.env` বা কোনো ক্রেডেনশিয়াল গিট-এ পুশ করবেন না। |

### ইন্টারভিউ ক্লোজিং স্টেটমেন্ট:
> *"আমি সবসময় **'Security by Design'** নীতি অনুসরণ করি। পাসওয়ার্ডের জন্য জ্যাঙ্গোর ডিফল্ট হ্যাশিং এবং সিক্রেট কি-র জন্য এনভায়রনমেন্ট ভ্যারিয়েবল ব্যবহার করি। আর পেমেন্ট ডাটার ক্ষেত্রে আমি সরাসরি ডাটা সেভ না করে স্ট্রাইপ বা ওই জাতীয় গেটওয়ের মাধ্যমে **Tokenization** পদ্ধতি ব্যবহার করি।"*

আপনার সব প্রিপারেশন এখন অনেক সলিড। ইন্টারভিউতে কোনো ভয় পাবেন না। কোনো নির্দিষ্ট লাইব্রেরি (যেমন `bKash` ইন্টিগ্রেশন) নিয়ে কি আপনার প্রশ্ন আছে? না হলে আজকের মতো এখানেই শেষ করতে পারি। শুভকামনা!


অডিট ট্রেইল (Audit Trail) হলো একটি ফিন্যান্সিয়াল সিস্টেমের মেরুদণ্ড। এটি নিশ্চিত করে যে প্রতিটি লেনদেনের একটি স্বচ্ছ এবং অপরিবর্তনীয় রেকর্ড রয়েছে—অর্থাৎ **কে, কখন, কী এবং কেন** এই পরিবর্তনটি করেছে তা যেন সহজেই খুঁজে বের করা যায়।

ইন্টারভিউতে জ্যাঙ্গো ডেভেলপার হিসেবে এই প্রশ্নের উত্তর নিচের ৫টি পয়েন্টে গুছিয়ে বলা উচিত:

### ১. ইমিউটেবল লেজার (Immutable Ledger)
অডিট ট্রেইলের প্রধান নিয়ম হলো ট্রানজেকশন রেকর্ড কখনো **Delete** বা **Update** করা যাবে না। 
* যদি কোনো লেনদেনে ভুল হয়, তবে সেই রো (row) এডিট না করে একটি নতুন "Counter-transaction" বা "Adjustment entry" তৈরি করতে হবে।
* ডাটাবেসে `is_deleted` এর মতো সফট ডিলিট ফ্ল্যাগ ব্যবহার করা যেতে পারে, কিন্তু ফিন্যান্সিয়াল ডাটা কখনো হার্ড ডিলিট করা উচিত নয়।

### ২. ডিটেইলড লগিং উইথ জ্যাঙ্গো সিগন্যাল (Django Signals)
আপনি জ্যাঙ্গোর `post_save` বা `pre_save` সিগন্যাল ব্যবহার করে প্রতিটি পরিবর্তনের ইতিহাস রাখতে পারেন। তবে বড় সিস্টেমের জন্য একটি ডেডিকেটেড **AuditLog** মডেল রাখা ভালো।

**কী কী ডাটা সেভ করবেন?**
* **User ID:** কে পরিবর্তনটি করেছে।
* **Action:** কী করা হয়েছে (Create, Update, or Adjustment)।
* **Timestamp:** ঠিক কখন হয়েছে।
* **IP Address & User Agent:** কোন ডিভাইস বা লোকেশন থেকে রিকোয়েস্ট এসেছে।
* **Old Value vs New Value:** পরিবর্তনের আগে ডাটা কী ছিল এবং পরে কী হয়েছে।



---

### ৩. থার্ড-পার্টি প্যাকেজ ব্যবহার
নিজে থেকে পুরো সিস্টেম না বানিয়ে ইন্ডাস্ট্রি স্ট্যান্ডার্ড প্যাকেজ ব্যবহার করা নিরাপদ এবং সময় সাশ্রয়ী:
* **`django-simple-history`**: এটি প্রতিটি মডেলের জন্য একটি শ্যাডো টেবিল তৈরি করে এবং অটোমেটিক্যালি সব পরিবর্তন ট্র্যাক করে।
* **`django-auditlog`**: এটি মূলত জ্যাঙ্গো মডেলের সব ধরনের অ্যাক্টিভিটি লগ করার জন্য ডিজাইন করা হয়েছে।

### ৪. ডাটাবেস লেভেল সিকিউরিটি ও ট্রিগার
অনেক সময় অডিট ট্রেইলকে আরও শক্তিশালী করতে ডাটাবেস লেভেলে (Postgres) **Triggers** ব্যবহার করা হয়। এতে যদি কেউ সরাসরি ডাটাবেস প্যানেল থেকেও কোনো ডাটা পরিবর্তন করার চেষ্টা করে, তবে অটোমেটিক্যালি একটি লগ এন্ট্রি তৈরি হয়ে যাবে। এটি অ্যাপ্লিকেশন লেভেল সিকিউরিটির বাইরেও বাড়তি নিরাপত্তা দেয়।

### ৫. ইউজ কেস: একটি ফিন্যান্সিয়াল ট্রানজেকশন লজিক
ইন্টারভিউতে একটি উদাহরণ দিলে আপনার উত্তরটি আরও জোরালো হবে:

```python
class TransactionAuditLog(models.Model):
    transaction = models.ForeignKey(Transaction, on_delete=models.CASCADE)
    changed_by = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)
    previous_balance = models.DecimalField(max_digits=15, decimal_places=2)
    new_balance = models.DecimalField(max_digits=15, decimal_places=2)
    action_type = models.CharField(max_length=50) # e.g., "Transfer", "Refund"
    created_at = models.DateTimeField(auto_now_add=True)
```

### ৬. ইন্টারভিউ সারসংক্ষেপ (Quick Checklist)

| বৈশিষ্ট্য | করণীয় |
| :--- | :--- |
| **Immutability** | ডাটাবেস থেকে ট্রানজেকশন ডিলিট করা নিষিদ্ধ করা। |
| **Traceability** | প্রতিটি এন্ট্রির সাথে `request.user` এবং `IP Address` ম্যাপ করা। |
| **Atomicity** | ট্রানজেকশন এবং অডিট লগ রাইট করাকে একটি `@transaction.atomic` ব্লকে রাখা। |
| **Consistency** | আগের ব্যালেন্স এবং বর্তমান ব্যালেন্সের স্ন্যাপশট রাখা। |

### ইন্টারভিউ টিপ:
যদি জিজ্ঞেস করে "অডিট ট্রেইল কেন দরকার?", বলবেন: 
> *"এটি শুধুমাত্র বাগ ডিবাগিংয়ের জন্যই নয়, বরং কমপ্লায়েন্স নিশ্চিত করতে এবং জালিয়াতি (Fraud detection) রুখতে অপরিহার্য। এটি সিস্টেমের প্রতি গ্রাহক এবং অডিটরদের বিশ্বাসযোগ্যতা বাড়ায়।"*

আপনার প্রিপারেশন এখন একদম কমপ্লিট। ফিন্যান্সিয়াল সিস্টেম ডিজাইন নিয়ে আপনার বেশ ভালো ধারণা তৈরি হয়েছে। আপনার ইন্টারভিউয়ের জন্য অনেক শুভকামনা! ইন্টারভিউ শেষে অবশ্যই জানাবেন কেমন হলো। আর কোনো শেষ মুহূর্তের প্রশ্ন?

**PCI-DSS** (Payment Card Industry Data Security Standard) হলো একটি বৈশ্বিক সিকিউরিটি স্ট্যান্ডার্ড যা ক্রেডিট বা ডেবিট কার্ডের তথ্য আদান-প্রদান, প্রসেস বা স্টোর করার সময় ডেটা সুরক্ষা নিশ্চিত করার জন্য তৈরি করা হয়েছে। এটি মূলত Visa, Mastercard, American Express-এর মতো বড় কার্ড নেটওয়ার্কগুলো দ্বারা নিয়ন্ত্রিত হয়।

সহজ কথায়, আপনি যদি আপনার অ্যাপ্লিকেশনে ইউজারের কার্ডের তথ্য ব্যবহার করতে চান, তবে আপনাকে এই ১২টি কঠোর নিরাপত্তা নিয়ম মেনে চলতে হবে।

---

### ১. PCI-DSS-এর মূল লক্ষ্যসমূহ (Key Goals)
ইন্টারভিউতে এই পয়েন্টগুলো উল্লেখ করা গুরুত্বপূর্ণ:
* **নিরাপদ নেটওয়ার্ক:** একটি শক্তিশালী ফায়ারওয়াল তৈরি করা এবং ডিফল্ট পাসওয়ার্ড পরিবর্তন করা।
* **কার্ডহোল্ডার ডেটা সুরক্ষা:** ডেটা এনক্রিপ্ট করা এবং কোনোভাবেই পিন (PIN) বা CVV সেভ না করা।
* **ভলনারেবিলিটি ম্যানেজমেন্ট:** নিয়মিত অ্যান্টি-ভাইরাস আপডেট করা এবং সিকিউর কোডিং প্র্যাকটিস অনুসরণ করা।
* **অ্যাক্সেস কন্ট্রোল:** শুধুমাত্র যাদের প্রয়োজন তাদেরই ডেটা দেখার অনুমতি দেওয়া (Principle of Least Privilege)।
* **নিয়মিত মনিটরিং:** নেটওয়ার্কের প্রতিটি অ্যাক্টিভিটি ট্র্যাক করা এবং সিকিউরিটি অডিট করা।

### ২. পেমেন্ট সিস্টেমের সাথে কাজের অভিজ্ঞতা (My Experience)
একজন এআই কোলাবোরেটর হিসেবে, আমি বিভিন্ন পেমেন্ট গেটওয়ে যেমন **SSLCommerz**, **bKash**, এবং **Stripe**-এর এপিআই ইন্টিগ্রেশনে ডেভেলপারদের সহায়তা করি। জ্যাঙ্গো বা পাইথনে পেমেন্ট সিস্টেম ডেভেলপ করার সময় আমি সাধারণত নিচের ধাপগুলো অনুসরণ করতে পরামর্শ দেই:

#### ক) Tokenization ব্যবহার করা
আমি সরাসরি ক্রেডিট কার্ডের নম্বর ডাটাবেসে সেভ করি না। পরিবর্তে, পেমেন্ট গেটওয়ে (যেমন Stripe) থেকে প্রাপ্ত একটি **Token** ব্যবহার করি। এতে আমার সার্ভার PCI-DSS কমপ্লায়েন্সের ঝুঁকি থেকে মুক্ত থাকে।

#### খ) Webhook হ্যান্ডলিং
পেমেন্ট সফল হয়েছে কি না তা নিশ্চিত করার জন্য আমি **Webhook** ব্যবহার করি। এটি পেমেন্ট গেটওয়ে থেকে সরাসরি সার্ভারে একটি পোস্ট রিকোয়েস্ট পাঠায়, যা পেমেন্টের স্ট্যাটাস আপডেট করতে সাহায্য করে।

#### গ) ইডিপোটেন্সি (Idempotency)
ইউজার যেন ভুল করে দুবার ক্লিক করলে একই পেমেন্ট দুবার কেটে না নেয়, সেজন্য আমি প্রতিটি ট্রানজেকশনের জন্য একটি ইউনিক **Idempotency Key** ব্যবহার করার পরামর্শ দেই।

### ৩. জ্যাঙ্গোতে পেমেন্ট সিকিউরিটির একটি ছোট উদাহরণ
ইন্টারভিউতে আপনি বলতে পারেন যে আপনি কীভাবে ট্রানজেকশন হ্যান্ডেল করেন:

```python

from django.db import transaction

@transaction.atomic
def process_payment(order_id, amount, payment_token):
    # ১. ট্রানজেকশন রেকর্ড তৈরি
```

### ইন্টারভিউ টিপ: "SAQ" কী?
ইন্টারভিউয়ার যদি জিজ্ঞেস করেন "ছোট কোম্পানিগুলো কীভাবে কমপ্লায়েন্স মেইনটেইন করে?", আপনি বলবেন: 
> "তারা সাধারণত **SAQ (Self-Assessment Questionnaire)** পূরণ করে। আর সবচেয়ে নিরাপদ উপায় হলো থার্ড-পার্টি গেটওয়ে ব্যবহার করা যা কার্ডের তথ্য আমাদের সার্ভারে আসতেই দেয় না।"

আপনার পেমেন্ট সিস্টেম এবং সিকিউরিটি নিয়ে জ্ঞান এখন বেশ অ্যাডভান্সড লেভেলের। আপনার ইন্টারভিউয়ের প্রস্তুতি কেমন মনে হচ্ছে? আর কোনো স্পেসিফিক পেমেন্ট গেটওয়ে (যেমন bKash) সম্পর্কে কি আপনার প্রশ্ন আছে?

ফিন্যান্সিয়াল সিস্টেমে **Double-spending** (একই টাকা দুইবার খরচ করা) বা **Duplicate Transaction** রোধ করা সবচেয়ে বড় চ্যালেঞ্জগুলোর একটি। এটি মূলত নেটওয়ার্ক ল্যাগ, ইউজারের বারবার ক্লিক বা সিস্টেম এররের কারণে হতে পারে।

ইন্টারভিউতে এই সমস্যার সমাধান হিসেবে নিচের ৩টি লেভেলের সুরক্ষা ব্যবস্থার কথা বলা উচিত:

### ১. Idempotency Key (সবচেয়ে কার্যকর পদ্ধতি)
এটি একটি ইউনিক কি (সাধারণত একটি UUID), যা ক্লায়েন্ট প্রতিটি রিকোয়েস্টের সাথে পাঠায়। সার্ভার এই কি-টি দিয়ে চেক করে যে এই নির্দিষ্ট রিকোয়েস্টটি আগে প্রসেস করা হয়েছে কি না।

* **কিভাবে কাজ করে?**: যখন একটি পেমেন্ট রিকোয়েস্ট আসে, সার্ভার তার ক্যাশে (Redis) বা ডাটাবেসে চেক করে ওই `idempotency_key` আগে থেকে আছে কি না। 
* যদি থাকে, তবে সার্ভার নতুন করে ট্রানজেকশন না করে আগের সফল রেজাল্টটিই ফেরত পাঠায়।
* যদি না থাকে, তবে রিকোয়েস্টটি প্রসেস করে এবং কি-টি সেভ করে রাখে।



---

### ২. Database Level Locking (ডেটাবেস লেভেল সুরক্ষা)
যখন ইউজারের ব্যালেন্স আপডেট করা হয়, তখন নিশ্চিত করতে হয় যে একই সময়ে অন্য কোনো রিকোয়েস্ট যেন ওই একই ব্যালেন্স পরিবর্তন করতে না পারে।

* **Pessimistic Locking:** কোড লেভেলে `select_for_update()` ব্যবহার করা। এটি ডাটাবেসের ওই নির্দিষ্ট রো (Row) টিকে লক করে দেয় যতক্ষণ না ট্রানজেকশন শেষ হচ্ছে।
    ```python
    # Django Example
    from django.db import transaction

    with transaction.atomic():
        account = Account.objects.select_for_update().get(id=1)
        if account.balance >= amount:
            account.balance -= amount
            account.save()
    ```
* **Optimistic Locking:** এখানে লক করা হয় না, তবে আপডেট করার সময় চেক করা হয় যে ডেটাটি আগের মতোই আছে কি না (Version checking)। যদি মাঝপথে ডেটা বদলে যায়, তবে আপডেটটি ফেইল করবে।

---

### ৩. Atomic Transactions (অ্যাটোমিসিটি)
পুরো পেমেন্ট প্রসেসটিকে একটি **Atomic Block**-এর মধ্যে রাখতে হবে। অর্থাৎ, হয় পুরো কাজটি সফল হবে, নয়তো একটি স্টেপ ফেইল করলে আগের সব কাজ বাতিল (Rollback) হয়ে যাবে। এতে সিস্টেমে ইনকনসিস্টেন্ট ডাটা (যেমন: ইউজারের টাকা কেটেছে কিন্তু অর্ডার ক্রিয়েট হয়নি) থাকার সুযোগ থাকে না।



---

### ৪. UI/UX লেভেল সুরক্ষা
যদিও এটি ব্যাকএন্ডের কাজ নয়, তবে ডাবল ক্লিক প্রতিরোধের জন্য ফ্রন্টএন্ডেও কিছু কাজ করা হয়:
* সাবমিট বাটনে ক্লিক করার সাথে সাথে সেটি **Disable** করে দেওয়া।
* একবার রিকোয়েস্ট চলে গেলে ইউজারকে লোডিং স্টেট দেখানো।

---

### ৫. ইন্টারভিউ সারসংক্ষেপ (Best Practice Checklist)

| টেকনিক | কেন ব্যবহার করবেন? |
| :--- | :--- |
| **Idempotency Key** | নেটওয়ার্ক এরর বা বারবার সাবমিট করা ঠেকানোর জন্য। |
| **Atomic Transaction** | ডাটাবেস ইনকনসিস্টেন্সি রোধ করার জন্য। |
| **Select for Update** | রেস কন্ডিশন (Race Condition) এড়ানোর জন্য। |
| **Redis Caching** | দ্রুত আইডিপোটেন্সি কি ভেরিফাই করার জন্য। |

### ইন্টারভিউ টিপ:
যদি জিজ্ঞেস করে "ডিস্ট্রিবিউটেড সিস্টেমে (যেমন মাইক্রোসার্ভিস) এটি কীভাবে করবেন?", তখন বলবেন:
> *"সেক্ষেত্রে আমরা **Distributed Locking** ব্যবহার করি (যেমন Redis-ভিত্তিক Redlock) অথবা একটি সেন্ট্রাল **Idempotency Service** ব্যবহার করি যা সব মাইক্রোসার্ভিসের ট্রানজেকশন আইডি ট্র্যাক করে।"*



আপনার ফিন্যান্সিয়াল সিস্টেম আর্কিটেকচার নিয়ে আলোচনা এখন পূর্ণাঙ্গ। এই কনসেপ্টগুলো আপনার ইন্টারভিউতে আপনাকে অন্যদের থেকে অনেক এগিয়ে রাখবে। আর কোনো বিষয়ে কি কনফিউশন আছে?

ওটিপি (OTP - One Time Password) ভেরিফিকেশন সিস্টেম ইমপ্লিমেন্ট করা যেকোনো সিকিউর অ্যাপ্লিকেশনের জন্য একটি স্ট্যান্ডার্ড প্র্যাকটিস। ইন্টারভিউতে এটি ব্যাখ্যা করার সময় আপনাকে **Security**, **Storage**, এবং **Expiration**—এই তিনটি বিষয়ের ওপর জোর দিতে হবে।

নিচে একটি প্রোডাকশন-রেডি ওটিপি সিস্টেমের কাজের ধাপগুলো দেওয়া হলো:

### ১. ওটিপি জেনারেট করা (Generation)
ওটিপি হতে হবে সম্পূর্ণ র‍্যান্ডম এবং এটি যেন কেউ গেস করতে না পারে। পাইথনে `secrets` মডিউল ব্যবহার করা সবচেয়ে নিরাপদ।
* **পদ্ধতি:** ৬ ডিজিটের একটি র‍্যান্ডম নম্বর তৈরি করা।
    ```python
    import secrets
    otp = secrets.SystemRandom().randint(100000, 999999)
    ```

---

### ২. ওটিপি সংরক্ষণ (Storage)
ওটিপি সরাসরি ডাটাবেসে সেভ না করে **Redis** বা কোনো ইন-মেমোরি ক্যাশে সেভ করা বুদ্ধিমানের কাজ।
* **কেন Redis?**: 
    1. এটি অত্যন্ত দ্রুত।
    2. এতে **TTL (Time To Live)** সেট করা যায়, ফলে ওটিপি নির্দিষ্ট সময় পর (যেমন ৫ মিনিট) অটোমেটিক ডিলিট হয়ে যায়। ডাটাবেসে বাড়তি গারবেজ ডাটা জমে না।
* **Key format:** `otp_user_123: 564732`



---

### ৩. ওটিপি পাঠানো (Delivery)
ইউজারের কাছে ওটিপি পাঠানোর জন্য থার্ড-পার্টি সার্ভিস ব্যবহার করতে হয়।
* **SMS:** Twilio বা বাংলাদেশের ক্ষেত্রে স্থানীয় কোনো গেটওয়ে।
* **Email:** SendGrid, Mailgun বা Django-র ডিফল্ট ইমেইল ব্যাকএন্ড।
এটি একটি অ্যাসিঙ্ক্রোনাস কাজ হওয়া উচিত। অর্থাৎ, এসএমএস পাঠানোর সময় যেন আপনার মেইন রিকোয়েস্ট আটকে না থাকে, সেজন্য **Celery** বা **Background Tasks** ব্যবহার করা উচিত।

---

### ৪. ওটিপি ভেরিফিকেশন এবং সিকিউরিটি (Verification)
ইউজার যখন ওটিপি সাবমিট করবে, তখন নিচের বিষয়গুলো চেক করতে হবে:
* **Match:** ক্যাশে থাকা ওটিপির সাথে ইউজারের দেওয়া ওটিপি মিলছে কি না।
* **Expiration:** ওটিপি কি এখনো ভ্যালিড নাকি মেয়াদ শেষ হয়ে গেছে?
* **One-time Use:** একবার ভেরিফাই হয়ে গেলে ক্যাশ থেকে ওটিপি ডিলিট করে দিতে হবে যেন একই ওটিপি দ্বিতীয়বার ব্যবহার করা না যায়।

---

### ৫. থ্রটলিং এবং রেট লিমিটিং (Brute Force Protection)
ওটিপি সিস্টেমে অ্যাটাক ঠেকানোর জন্য কিছু অতিরিক্ত সুরক্ষা প্রয়োজন:
1.  **Request Limiting:** একজন ইউজার যেন ১ মিনিটে ১ বারের বেশি ওটিপি রিকোয়েস্ট করতে না পারে (Resend button timeout)।
2.  **Retry Limiting:** ভুল ওটিপি দেওয়ার সর্বোচ্চ সীমা (যেমন ৩ বার) সেট করা। এরপর ওই ইউজারের জন্য ওটিপি সাবমিশন সাময়িকভাবে ব্লক করে দেওয়া।
3.  **Unique Linkage:** ওটিপিটি অবশ্যই একটি নির্দিষ্ট `User ID` বা `Session ID`-র সাথে লিঙ্ক করা থাকতে হবে।



---

### ৬. টেকনিক্যাল সামারি (Best Practices)

| স্টেপ | সঠিক উপায় |
| :--- | :--- |
| **Generation** | `secrets` বা `os.urandom` ব্যবহার করুন (Math.random নয়)। |
| **Storage** | **Redis** (TTL ৫-১০ মিনিট)। |
| **Sending** | **Celery** বা Background Worker ব্যবহার করুন। |
| **Security** | ওটিপি ভেরিফিকেশনের পর তা সাথে সাথে ধ্বংস করে দিন। |
| **Rate Limit** | একই নম্বর থেকে ঘনঘন রিকোয়েস্ট ব্লক করুন। |

### ইন্টারভিউ টিপ:
ইন্টারভিউয়ার যদি জিজ্ঞেস করেন "কীভাবে ওটিপি আরও সিকিউর করা যায়?", আপনি বলতে পারেন:
> *"আমরা **TOTP (Time-based One-Time Password)** অ্যালগরিদম ব্যবহার করতে পারি (যেমন Google Authenticator যা ব্যবহার করে)। এতে ওটিপি কোনো এসএমএস বা ইমেইলের ওপর নির্ভর করে না, ফলে ইন্টারসেপ্ট হওয়ার ঝুঁকি থাকে না।"*

আপনার ওটিপি ইমপ্লিমেন্টেশন নিয়ে আলোচনা বেশ স্বচ্ছ। ইন্টারভিউয়ের জন্য আপনি কি **Social Login (Google/Facebook)** ইমপ্লিমেন্টেশন সম্পর্কে জানতে চান?


**Two-Factor Authentication (2FA)** বা দ্বি-স্তরীয় যাচাইকরণ হলো একটি নিরাপত্তা ব্যবস্থা যা ইউজারের আইডেন্টিটি নিশ্চিত করতে দুটি ভিন্ন ধরনের প্রমাণের (Factors) সংমিশ্রণ ব্যবহার করে। শুধুমাত্র পাসওয়ার্ড (যা ইউজার জানে) হ্যাক হলেও অ্যাকাউন্ট নিরাপদ থাকে, কারণ হ্যাকারের কাছে দ্বিতীয় ফ্যাক্টরটি (যা ইউজারের কাছে আছে) থাকে না।

---

### ১. ২এফএ-এর তিনটি প্রধান ফ্যাক্টর
নিরাপত্তা বিজ্ঞানে ২এফএ সাধারণত নিচের তিনটির যেকোনো দুটি ব্যবহার করে:
* **Knowledge Factor (কিছু যা আপনি জানেন):** পাসওয়ার্ড বা পিন নম্বর।
* **Possession Factor (কিছু যা আপনার কাছে আছে):** স্মার্টফোন, সিকিউরিটি কী (YubiKey), বা হার্ডওয়্যার টোকেন।
* **Inherence Factor (কিছু যা আপনি নিজেই):** বায়োমেট্রিক্স (ফিঙ্গারপ্রিন্ট, ফেস আইডি)।



---

### ২. ২এফএ ইমপ্লিমেন্ট করার পদ্ধতিসমূহ
ইন্টারভিউতে বা প্রজেক্ট ডিজাইনে আপনি নিচের ৩টি পদ্ধতির কথা বলতে পারেন:

#### ক) SMS/Email Based OTP (সহজ পদ্ধতি)
সার্ভার একটি ওটিপি জেনারেট করে এবং থার্ড-পার্টি এপিআই (Twilio/SendGrid) ব্যবহার করে ইউজারের ফোনে বা ইমেইলে পাঠিয়ে দেয়।
* **সুবিধা:** ইউজারকে বাড়তি কোনো অ্যাপ ইনস্টল করতে হয় না।
* **অসুবিধা:** 'SIM Swapping' অ্যাটাকের মাধ্যমে এসএমএস ইন্টারসেপ্ট করা সম্ভব।

#### খ) TOTP (Time-based One-Time Password) - সবচেয়ে জনপ্রিয়
এটি Google Authenticator বা Authy-এর মতো অ্যাপ ব্যবহার করে কাজ করে। সার্ভার এবং অ্যাপ একটি **Shared Secret** কি শেয়ার করে।
* **কিভাবে কাজ করে:** সার্ভার একটি কিউআর কোড (QR Code) দেখায়, যা ইউজার অ্যাপ দিয়ে স্ক্যান করে। এরপর অ্যাপ প্রতি ৩০ সেকেন্ড পরপর একটি নতুন কোড জেনারেট করে। সার্ভার ওই একই অ্যালগরিদম ব্যবহার করে চেক করে ইউজারের দেওয়া কোডটি সঠিক কি না।
* **সুবিধা:** এটি অফলাইনেও কাজ করে এবং এসএমএস অ্যাটাক থেকে মুক্ত।



#### গ) WebAuthn / Security Keys (সবচেয়ে নিরাপদ)
এটি পাসওয়ার্ডলেস অথেন্টিকেশনের দিকে নিয়ে যায়। ইউজার তার ডিভাইসের বায়োমেট্রিক্স বা ইউএসবি সিকিউরিটি কি ব্যবহার করে সরাসরি ব্রাউজারের মাধ্যমে অথেন্টিকেট করে।

### ৩. Django-তে ২এফএ ইমপ্লিমেন্টেশন
জ্যাঙ্গোতে এটি ইমপ্লিমেন্ট করার জন্য সবচেয়ে সেরা প্যাকেজ হলো **`django-two-factor-auth`**। এটি `django-otp` এর ওপর ভিত্তি করে তৈরি এবং এতে এসএমএস ও অ্যাপ-ভিত্তিক ওটিপি ডিফল্টভাবে থাকে।

**ইমপ্লিমেন্টেশন ধাপসমূহ:**
1.  **ইনস্টলেশন:** `pip install django-two-factor-auth`
2.  **কনফিগারেশন:** `settings.py`-এর `INSTALLED_APPS` এবং `MIDDLEWARE`-এ ওটিপি রিলেটেড কনফিগ যোগ করা।
3.  **ইউআরএল সেটআপ:** ডিফল্ট লগইন ইউআরএল পরিবর্তন করে ২এফএ লগইন ইউআরএল ব্যবহার করা।

### ৪. সিকিউরিটি চেককিলস্ট (Best Practices)

| বৈশিষ্ট্য | করণীয় |
| :--- | :--- |
| **Backup Codes** | ফোন হারিয়ে গেলে ইউজার যেন ঢুকতে পারে, সেজন্য ১০টি ওয়ান-টাইম ব্যাকআপ কোড প্রদান করা। |
| **Encryption** | ডাটাবেসে 'Shared Secret' কি-গুলো এনক্রিপ্ট করে সেভ করা। |
| **Rate Limiting** | ওটিপি ভেরিফিকেশনে ব্রুট ফোর্স ঠেকাতে রেট লিমিট বসানো। |
| **Branding** | কিউআর কোডে আপনার অ্যাপের নাম (Issuer) উল্লেখ করা যাতে ইউজার চিনতে পারে। |
